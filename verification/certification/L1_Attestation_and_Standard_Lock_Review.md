## 2. Standard Lock Review ‚Äî synthesis

**The Lock question:** *Can independent teams implement + interoperate + be cross-certified from the frozen corpus + ratified ACS today?*

### **Lock verdict: CONDITIONAL**

The honest answer is CONDITIONAL ‚Äî trending toward YES on the byte-exact identity surface, but **not yet achievable today** because (a) the ACS set is **drafted, not CCP-ratified** (carries no conformance weight), (b) the **conformance suite is not yet populated** and no differential/divergence-detection harness exists, (c) the **reference runtime has not adopted ACS** (the standard is paper, not proven-runnable), (d) one **live interop-breaking domain-tag collision** sits inside the batch, and (e) coverage is **one type** with **no trace/replay standard**.

### Lock rationale ‚Äî what is genuinely strong (why this is not NO)

The Lock is **not NO** because the byte-exact interop surface is now specified well enough that convergence is achievable *in principle* ‚Äî a real advance over the prior NOT-READY state:

- **ACS-001/002/003 are independently reproducible.** A source-blind team rebuilt a dCBOR encoder + multihash addresser from the normative text alone and re-ran every published vector: **28/28 byte-for-byte** (ContentIds, canonical bodies, the 327-byte envelope, domain-tag separation, encoded-key sort, exact-i64-ns timestamps via dCBOR-over-JCS). This is the operational definition the corpus itself demands ("shared vectors, not prose") and it passes for the pinned surface.
- **ACS-004 closes the empty-type-registry keystone for one type** (`uci.fact@1.0`): closed schema-document shape, closed type-code + cardinality vocabulary, deterministic 6-clause validator, aspect‚Üícarrier binding, byte-exact 430-B schema body + 264-B instance body + SHA-256 ContentIds (independently OpenSSL-verified).
- **The reference commit gateway is real and disciplined:** `RefKernel::commit` (ProposedWrite‚ÜíWAL‚ÜíTruthRef) is content-idempotent (ORCH-004) and replayable (ORCH-003), with LAYER-001/OWN-001 mechanically gated.

If those bytes are faithfully implemented, independent teams **would** agree on identity, serialization, envelope, and typed `uci.fact` instances for the covered cases.

### Lock rationale ‚Äî why it is only CONDITIONAL (the residual gaps)

1. **No executable differential / divergence-detection harness (decisive ‚Äî Global Readiness R-07 / CCG-6).** `arves-conformance` is an explicit no-implementation skeleton; verdicts are property-only with "no expected output," which by design cannot catch byte-level divergence in types/envelopes/digests (two wire-incompatible runtimes could both PASS). **Independently confirmed:** none of the published ContentIds (`4284f0ac`, `56e30f71`, `6b3f99c6`, `fc0ef055`) is asserted by any `.rs` test; `verification/reproducibility/` does not exist; `verification/{certification,independent}/` are one-line README stubs. **Today, nothing would detect divergence.**
2. **Reference runtime has not adopted ACS.** `arves-kernel` still defines `ContentHash(pub Vec<u8>)` as caller-supplied with "byte layout intentionally unspecified," and `payload` is opaque bytes; no dCBOR codec / multihash construction / ACS-004 validator exists in any crate. **Independently confirmed** via source read + grep. Without an ACS-computing reference, the interop claim is *unfalsified* (ACS-004 ¬ß14 + the Consistency Report both list adoption as OPEN).
3. **Standards are DRAFT, not Ratified ‚Äî and contain a live domain-tag collision.** **Independently confirmed:** all of ACS-001..005 are stamped `Status: DRAFT (Candidate on CCP-GATE pass; not yet Ratified)` ‚Äî none has passed CCP-GATE, so none is a registered normative addition an independent team may bind to. More seriously, the Consistency Report caught + fixed the ACS-003‚ÜîACS-004 `0x06` collision but **missed that ACS-005 ¬ß8 re-allocates the same tags for a third meaning**: ACS-005 ¬ß8 assigns `0x06` = normative-glossary term-set and `0x07` = requirement clause, colliding with ACS-003's `0x06` (canonical-envelope) and ACS-004's `0x07` (type-schema). The `CCP-005-CS-1` vectors bake this in at the byte level (vector 1 pre-image begins `06‚Ä¶`, vector 2 begins `07‚Ä¶`). Since ACS-001 is declared the single domain-tag registry authority, this is exactly the silent-interop-break class the batch exists to prevent ‚Äî now present **inside** the batch and unreconciled.
4. **Coverage is one type; the trace/replay format is absent.** ACS-004 fixes only `uci.fact@1`; the other ~17 `uci.*` root types + aspect/relation schemas (the separate ACS-005-Schema-Registry work) are unpopulated, so teams cannot yet interoperate on `uci.goal`, `uci.observation`, etc. R-04/CCG-4 ‚Äî the Decision-Trace + Runtime-Fingerprint schema for ORCH-003 cross-runtime replay, the heart of portable certification ‚Äî has **no standard at all** in this batch. No populated Event-Catalog payload schema exists (ACS-003 defines only the envelope).
5. **Type-layer + ontology are not wired from the frozen corpus alone.** From the frozen Ontology Spec + *registered* invariants only (excluding drafts), a team cannot construct/validate/address a `uci.fact`: Table 2 is 18 URNs with one-line glosses, Part 4 names five aspects with no field list/type/cardinality/encoding, and the `schema` slot is reserved-and-empty (P04-F1 / R-03). Missing normative API signatures: no `EntityUrn` parse/format grammar (the runtime invented `urn:arves:<ns>:<type>@<maj>.<min>:<local>`), no `validate(instance, schema)` signature, no schema-document encoder / registry-population API, no instance‚Üí`ProposedWrite` adapter. `G-001`/`QUERY-001` remain PROPOSED/informative yet the Kernel doc-comments lean on them.

### Residual gaps that must close before "10 teams" (cross-certification real)

To move the Lock from CONDITIONAL ‚Üí YES (the Manual Part 13 A/B goal, generalized to 10 independent runtimes):

- **G1 ‚Äî Ratify the ACS batch through CCP-GATE.** ACS-001..005 must pass CCP-GATE (each with its conformance scenario) before it is a registered normative addition. *Blocking for any binding claim.*
