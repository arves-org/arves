## Design resolutions (open points; simplest option consistent with IDR-001..005 — recorded, never silently invented)

| # | Resolution |
|---|---|
| DR-1 | **No promotion/move of RCR-010:** the design lists `QueryProjection`/`QueryProbe` as "reused, **unmodified**" conformance reference semantics (§2) and prescribes I3 runtime code as "additive `arves-query` impls" (§6.2 row 1). Therefore `arves-conformance` is UNTOUCHED by this RCR; the query core is a fresh implementation inside `arves-query` honoring RCR-010's semantics (replay from the retained log, no Kernel read). |
| DR-2 | **Contract `ShardKey` (opaque String) text form:** canonical `"tenant/workspace"` (the A-004 key), parsed at the FIRST `/`; unparseable or unknown → `UnknownShard`. A tenant containing `/` is not routable via this text form in the Stage-1 core — and the limitation is stronger than mere unroutability: `arves-kernel::ShardKey::new` permits `/` inside parts, so two DISTINCT committed shards (tenant=`a/b`, ws=`c`) and (tenant=`a`, ws=`b/c`) alias to the same text `a/b/c`, which always resolves to `(a, b/c)` — a caller intending tenant `a/b` is silently served tenant `a`'s shard if it exists (a **misdirected read**, not a cross-shard leak: WAL keys stay distinct and the fold never mixes records; v1.0's trusted-single-host model already lets any caller name any shard). Longer-term: the typed contract key already noted here, or an RCR rejecting `/`-bearing parts in `ShardKey::new`. |
| DR-3 | **OQ-7 (payload shape):** resolved to the design's named safe default — `Query::View = Vec<u8>` raw committed payload bytes, never interpreted; `ProjectionId` = lowercase hex of the record's `ContentId` (content-addressable, ORCH-004). |
| DR-4 | **Version vocabulary:** `Projection.observed_at` = the fold's **trace position** (offsets `[0..observed_at)` applied — the point the read reflects, §3.11 reproducibility); `latest_version(id)` = the **commit offset** of the entry's current value (matches the Kernel's `CommitIndex` for the same record), always < `observed_at`. Both documented on the API. |
| DR-5 | **Fold rule:** only `RecordKind::Outcome` materializes an entry; `Membership`/`SnapshotMarker`/`Barrier` advance the position only (they carry no truth). A foreign-shard record never enters the fold and never advances it (structural SHARD-001). Duplicate content ids resolve last-write-wins by offset (the Kernel gateway already dedupes at commit, ORCH-004, so kernel-fed logs contain no duplicates). |
| DR-6 | **Single-node tier semantics (honest, §3.5 step 3 degenerate):** Linearizable = catch the fold up to the local WAL head then serve — the degenerate of IDR-001 read-index (on one node the local committed log IS the commit index; no remote leader exists). BoundedStaleness = catch up then serve: the core proves `lag 0ms ≤ bound` by being current, so `StalenessBoundExceeded` is **unreachable in this core by construction** — the distributed follower lag-attestation (time↔index) is OQ-2 and awaits its own IDR. Eventual = serve the standing fold WITHOUT refresh (observably stale, never wrong for its `observed_at`). `served_tier` always == requested, never stronger. |
| DR-7 | **Compacted prefix / durable faults / out-of-range version:** a shard whose retained log cannot reproduce the full fold (`earliest > applied`, Stage 1 has no query-side snapshot bootstrap) or whose replay errors is refused as `UnknownShard` ("not servable by this replica") — the least-wrong member of the frozen error enum; a richer read-fault variant and the §3.5-step-1 snapshot bootstrap are future RCR surface (the Kernel's snapshot blob is Kernel-encoded state the Query layer must not interpret). `at_version` beyond the committed head fails loudly (assert) — lossless-or-loud, never a fabricated trace point. |
| DR-8 | **Dependency edges:** exactly ONE new architectural edge, `arves-query`(60) → `arves-persistence`(20) (downward, LAYER-001; design §3.4 row 1). NO edge to kernel (no Kernel read hook — ORCH-001), consensus (no distributed surface yet), or LCW (OQ-8 unresolved; no LCW read views claimed). `arves-kernel` is dev-only for tests. |
| DR-9 | **Bootstrap-then-lag:** the first read touching a shard bootstraps its fold to the then-current head; Eventual reads thereafter serve without refresh. This models replica bootstrap + lag deterministically with zero clocks and makes tier staleness observable in tests. |

## HONEST SCOPE — what this is and is not

- **Is:** the single-node read path of I3 — ONE process reading ITS OWN store by WAL
  replay. Truth remains CP (only the Kernel commits); this surface is the AP observability
  side and *labels* what it serves (`served_tier`, `observed_at`) — it adds no consistency
  and subtracts none (design §3.12).
- **Is NOT (nothing distributed is claimed):** no query routing fabric, no replica sets,
  no follower reads, no real read-index leader round-trip, no staleness attestation
  (OQ-2 IDR pending), no scatter-gather (additive contract surface per design §3.3/§6.2),
  no LCW read views (OQ-8), no network, no authN/authZ (OQ-1 — trusted single host, as at
  v1.0). "Leader-read vs follower-read staleness" does not yet exist here: there is one
  replica, and the only staleness is the deliberate not-refreshed Eventual fold.
- **QUERY-001 remains PROPOSED** — cited only as proposed; the read-only obligation is
  enforced via the registered A-003 Layer-Matrix row and the frozen trait shape
  (type-level: every method `&self`; the module consumes the WAL behind `&` so the
  `&mut` write surface is uncallable; the crate has no kernel dependency at all).

## Invariants (registered only)

- **OWN-001** — Query owns only the derived, disposable fold; the store of record stays the
  Kernel-committed WAL (rebuild-equality + no-state-change tests).
- **LAYER-001** — one new downward edge (60→20); architecture gate green; no upward or
  lateral edge; dev-dep excluded by the gate's own rule.
- **SHARD-001** — every read resolves exactly one `(tenant, workspace)` shard; fold-level
  structural isolation; two-tenant negative proof on every tier.
- **ORCH-001** — reads reconstruct committed truth from the WAL only; no Kernel read hook;
  nothing staged/uncommitted can be seen (only committed records exist in the WAL).
- **ORCH-003** — pure-fold determinism: replica-equality, digest == truth-hash basis
  (incl. across Kernel recover), snapshot-at-index equality, checkpoint⊕suffix ≡ full replay.
- **ORCH-004** — content-addressable `ProjectionId` (hex of `ContentId`); idempotent reads
  (identical results, zero state change anywhere).
- ORCH-002 — untouched (no Control-Plane surface in this stage).
