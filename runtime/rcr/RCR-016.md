## The gap (product need)

Only the capability pre-bound at bridge startup (`derive.fact` in the default `t1/w1`
shard) could ever be invoked. With RCR-014 opening non-default shards, every fresh shard
started permanently unusable for `invoke` (honest `ERR unbound`, by RCR-014's documented
no-auto-bind choice) â€” and there was no protocol verb to bind anything. Products could
not name their own capabilities or enable invocation in their own shard without
recompiling the bin.

## The change (additive)

**Server** (`crates/arves-bridge/src/bin/arves-bridge.rs`):
- New verb `bind <capability>`, composable with the `id=` / `shard=` prefixes
  (RCR-011/RCR-014). It registers and binds `<capability>` in the target shard to the
  reference engine identity `engine:derive.fact@1.0.0`, under the reference invocation
  contract. Response: `bound <capability>`.
- **Idempotent rebind:** binding a name already bound in that shard answers `bound`
  again and changes nothing (every binding in this bin points at the one reference
  engine, so there is nothing to supersede â€” no `NonMonotonicVersion` surprise).
- Malformed bind (missing name, extra tokens, name > 64 bytes) â†’ `ERR bad-request`.
  A whitespace-bearing name is unrepresentable (tokenization splits it â†’ extra tokens).
- `main`'s built-in default binding and the `bind` verb now share ONE helper
  (`bind_reference_capability`), so a dynamically bound name behaves exactly like the
  built-in one by construction. `handle_request`/`respond` take `&mut` registry;
  commit/invoke never mutate it.

**Client** (`products/arves-sdk-ts/src/bridge.mjs` â€” living, product-side):
- New `bind(capability)` method: validates the token (same interpolation rule as
  `invoke`), sends `bind <capability>`, verifies the `bound <capability>` response
  shape, returns `{ bound }`. Throws a clean typed error on `ERR`.

## HONEST SCOPE (stated in the bin header and the client doc)

This bin hosts exactly **one** engine â€” the reference `engine:derive.fact@1.0.0`.
`bind` attaches capability **names** to that one engine identity; it does **not** load
or execute arbitrary engine code. Dynamic engine loading is not a bridge feature (and
would be an entirely different RCR class).

Confusability note (review finding, comment-only): capability names are fully **opaque**
tokens (the pre-existing `invoke` treatment), so a name may visually mimic a protocol
token â€” `bind shard=x/y` lawfully binds the name `shard=x/y`. No ambiguity: the
`id=`/`shard=` prefixes are recognized positionally (leading tokens only), never by the
content of a name position. Stated in the bin header; no code change.
