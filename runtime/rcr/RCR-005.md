# RCR-005 — Kernel commit-gateway content-integrity (Runtime v1.1)

**Status:** Applied (v1.1) · **Type:** additive (new `CommitError` variant + gateway check +
test) · **Gate:** `cargo test --workspace` → **78 passed / 0 failed** (was 77; +1
content-integrity test). Freeze baseline re-advanced via `freeze_check.py update`.

## Motivation (SYSTEM_GAP_ANALYSIS #3)

The Kernel is the **sole commit gateway** for cognitive truth (OWN-001). It makes commits
idempotent by `ContentHash` equality (ORCH-004): a re-proposal bearing an already-committed
`ContentHash` resolves to the existing `TruthRef` (`AlreadyCommitted`). But the pre-RCR gateway
did **not** check that the re-proposal's *payload* matched the one already bound to that address —
so a caller could commit `payloadA` under `hashX`, then submit `payloadB` under the **same**
`hashX` and receive a silent idempotent no-op. That breaks the load-bearing invariant *address ⇒
content*: ORCH-004 idempotency is only sound if a content address denotes exactly one content.

## Change (what shipped)

- New `CommitError::ContentIntegrity { shard }` variant + its `Display`.
- `RefKernel::commit`: on the idempotent path, if the incoming `payload` differs from the payload
  already bound to that `(shard, ContentHash)`, reject with `ContentIntegrity` instead of silently
  returning the prior truth (and never fork — `committed_count` is unchanged). An *identical*
  re-proposal is still an idempotent `AlreadyCommitted` no-op (unchanged).
- Test `behaviour_7_content_integrity_same_address_different_payload` (walking_skeleton): same
  address + same payload → `AlreadyCommitted`; same address + **different** payload →
  `ContentIntegrity`; the mismatched fork is not committed.

## Layering decision — why this is NOT an ACS-001 recompute (NON-NEGOTIABLE RULE #9)

Gap #3's fuller phrasing is "the Kernel should compute/verify the ACS-001 multihash at the
gateway." Doing **that** would require the Kernel to recompute `0x12 0x20 ‖ SHA-256(domain ‖
payload)` — which needs (a) a **`domain`** on `ProposedWrite` (an ABI change) and (b) a Kernel →
`arves-acs` **dependency**. The Kernel is deliberately ACS-agnostic: `ContentHash`'s byte layout is
"intentionally unspecified" and `arves-kernel` depends only on `arves-persistence`. Coupling the
Kernel to the ACS codec to re-derive an address is precisely what **CLAUDE.md NON-NEGOTIABLE RULE
#9 ("Never couple runtime components unnecessarily")** forbids, and the layered-correct owner of
ACS-001 addressing is the **bridge / Information Platform** (where address integrity is already
enforced — `arves-bridge`). So RCR-005 enforces the half of #3 the Kernel legitimately owns — the
`ContentHash ⇒ payload` binding, using only Kernel state — and **does not** couple the Kernel to
ACS-001. Whether to move full multihash re-derivation into the Kernel (accepting the coupling +
ABI change) is a maintainer architecture decision, recorded here, **not** taken in this RCR.

## Scope (honest)

- **IS:** the Kernel now rejects the "same address, different content" fork at the sole gateway,
  closing the Kernel-owned half of #3 with no new dependency and no ABI change to `ProposedWrite`.
- **IS NOT:** ACS-001 multihash recomputation at the Kernel (deferred by the RULE #9 decision
  above; address integrity remains enforced at the bridge). No `standard/` byte touched.
- **Independence unchanged: G1.**
