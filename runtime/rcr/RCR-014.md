# RCR-014 — Bridge per-request shard selection (v1.1)

**Status:** APPLIED (2026-07-05) · **Class:** additive, backward-compatible (v1.1)

## The gap (product need)

The `arves-bridge` bin pinned every request to a hard-coded `ShardKey { t1, w1 }`. A
product that needs tenant isolation (SHARD-001 is *the* multi-tenant invariant) was
forced into **process-per-tenant**: one bridge process per shard, one Kernel per
process — no way to observe two shards' distinct truths through one session. The
runtime underneath (MemKernel, MemRegistry) is already fully shard-scoped; only the
protocol seam withheld it.

## The change (additive)

**Server** (`crates/arves-bridge/src/bin/arves-bridge.rs`):
- After the optional `id=` token (RCR-011), a request line MAY carry
  `shard=<tenant>/<workspace>` — each part non-empty, whitespace-free, ≤ 64 bytes,
  exactly one `/`. The commit/invoke runs in THAT shard. Applies to commit AND invoke
  lines. Token order is fixed: `id=` first (if present), then `shard=`; shard-only is
  lawful.
- Absent token → default `t1/w1`, **byte-identical** to before (backward compatible).
- Malformed spec → `ERR bad-shard` WITHOUT reflecting the untrusted spec back
  (RCR-011 bad-id discipline); a *valid* id prefix is still echoed for correlation.
- **Documented choice:** `invoke` in a non-default shard where the capability was never
  bound is honestly refused `ERR unbound` — NO implicit auto-bind (an implicit bind
  would silently mint capability configuration as a side effect of an invoke). The
  explicit `bind` verb (RCR-016) is the lawful way to bind a capability in a new shard.
- `respond` now performs id split → shard split → handle; `main` no longer holds a
  fixed shard.

**Client** (`products/arves-sdk-ts/src/bridge.mjs` — living, product-side):
- `KernelBridge` options gain `{ tenant, workspace }` (both together); when given,
  every request carries the `shard=` token. Parts validated client-side (non-empty, no
  whitespace or `/`, ≤ 64 bytes) with a clean typed error. Default: no token — bytes
  on the wire identical to before.

## Proof

- 4 new server protocol tests (`rcr014_*`) at the `respond()` level: same body in two
  shards → two distinct truths with distinct index sequences and idempotency scopes
  (SHARD-001 observable — the fresh `committed` in shard 2 bites if the token were
  ignored); id/shard token ordering incl. wrong-order refusal; malformed shard specs
  refused (`ERR bad-shard`, id still correlated, MAX boundary accepted); invoke in an
  unbound shard refused `ERR unbound`. `cargo test --workspace` 98 → **102 / 0**.
- 2 new product regressions (`products/robustness.test.mjs`, 50 → **52/52**), both over
  the REAL exe: (a) raw-protocol — same body in `t1/w1` and `acme/hq` is fresh truth
  in each with per-shard idempotency; (b) client-level — a shard-scoped `KernelBridge`
  commits fine and its `invoke derive.fact` is refused `unbound` (proves the token is
  actually sent AND honored; would pass vacuously under neither).

## Honest boundary

- Shard selection is **per request over one in-memory Kernel session**; this is
  multi-tenant *protocol* access, not distributed shard placement (per-shard Raft is
  I2+ work, IDR-002/IDR-004 untouched).
- The reference capability `derive.fact` remains bound only in the default shard;
  non-default shards start empty by design (see documented choice above).

## Freeze discipline

`runtime/` bytes changed under this RCR (bridge bin only; the frozen `Kernel` /
`CapabilityRegistry` traits and lib.rs are untouched). Freeze-manifest re-baseline is
left to the maintainer integration loop (this change set deliberately does not run
`freeze_check.py update`). Recorded in `RUNTIME_FREEZE_v1.0.md` (applied-RCRs).
