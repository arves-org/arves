# RCR-017 ‚Äî Opaque ShardKey (SHARD-001-F2) (v1.1)

**Status:** APPLIED (2026-07-05) ¬∑ **Class:** workspace-internal breaking refactor,
**wire-compatible** (v1.1) ‚Äî no external Rust consumers exist (products speak the line
protocol, IDR-006), so the Rust-surface break is lawful under this RCR; the bridge line
protocol and every response byte are unchanged.

## The gap (Madde-11 audit finding SHARD-001-F2)

SHARD-001 declares the shard key **immutable** ‚Äî but `arves-kernel::ShardKey` (and the
capability fabric's mirror) exposed `pub tenant` / `pub workspace` fields: any holder
could mutate a part in place or construct a degenerate key (empty tenant), making the
invariant a convention instead of a property of the type. The opaque
`arves-invariants::ShardKey` existed but was used by nothing. Tracked as SHARD-001-F2 in
`verification/OPEN_DEBT_REGISTER.md` (open since the audit).

## The change

**`arves-kernel`** (`crates/arves-kernel/src/lib.rs`):
- `ShardKey` fields are now **private** ‚Äî immutable *by type*.
- Sole constructor `ShardKey::new(tenant, workspace) -> Result<Self, ShardKeyError>`:
  rejects an **empty** part (`EmptyTenant`/`EmptyWorkspace`) and a part over
  **`ShardKey::MAX_PART_BYTES` = 256** bytes (`TenantTooLong`/`WorkspaceTooLong`).
- Read-only accessors `.tenant()` / `.workspace()`. `ShardKeyError` implements
  `Display` + `Error`. Crate-internal code (WAL mapping `to_pshard`/`from_pshard`,
  error `Display`s, `truth_hash`, `commit_inner`) keeps direct field access ‚Äî lawful
  only inside the crate, documented on `from_pshard` (replayed keys were written under
  constructor-validated keys).

**`arves-capability-fabric`** ‚Äî **aligned the same way** (the offered alternative was
documenting why not; alignment wins because the fabric key partitions binding state
under the same SHARD-001 and had the identical mutable-by-type hole): private fields,
`ShardKey::new` with the IDENTICAL rules (so kernel‚Üífabric key conversion at the bridge
seam is total), accessors, its own std-only `ShardKeyError` (this crate depends on no
sibling crate per its LAYER-001 note ‚Äî the small validation duplication is documented
in the type's rustdoc).

**Every in-workspace call site updated** (workspace-internal breaking refactor):
kernel tests (`walking_skeleton`, `recovery`, `persistent_wal`, `checkpoint`),
`arves-runtime` demo/harness bin, `arves-conformance::live` (probe shards + the
reference `Connector`, which now `expect`s well-formed Source scoping; the SHARD-001
probe reads via accessors), `arves-bridge` lib (invoke seam) + bin (`default_shard`,
the RCR-014 `shard=` parser ‚Äî its ‚â§64-byte grammar is strictly tighter than the
constructor's ‚â§256, so `ERR bad-shard` behaviour is unchanged ‚Äî and the RCR-016 `bind`
arm). Kernel‚Üîfabric conversions carry an invariant-stating `expect` (identical rules ‚áí
total). NOT touched: `arves-persistence`/`arves-lcw`/`arves-ontology`/
`arves-execution`/`arves-control-plane`'s own key types (different layers, different
owners ‚Äî widening the scope would be invariant-enforcement by stealth), and the unused
opaque `arves-invariants::ShardKey` (consolidating it into the kernel would be a new
layering decision, honestly left open in the debt register).

## Proof (tests that bite)

- `behaviour_10_degenerate_shard_key_unrepresentable` (kernel integration test ‚Äî from
  OUTSIDE the crate, which is the point): `ShardKey::new("", "w1")` ‚Üí
  `Err(EmptyTenant)`; empty workspace, >256-byte parts refused with the exact lengths;
  the 256 boundary accepted; accessors round-trip; a valid opaque key still commits
  truth through the unchanged gateway.
- `rcr017_degenerate_shard_key_unrepresentable` (capability-fabric): same constructor
  refusals for binding partitions.
- Field mutation is now a COMPILE error outside the owning crates (unrepresentable, not
  tested at runtime ‚Äî the type system is the test).
- `cargo test --workspace` 108 ‚Üí **110/0** (every pre-existing kernel/persistence/
  bridge/conformance behaviour proof green over the opaque key).
- Product suite `products/robustness.test.mjs` **55/55** on the rebuilt exe ‚Äî the line
  protocol (incl. RCR-014 `shard=` and RCR-015 `--wal-dir` paths) is byte-identical.

## Honest scope

This closes the *mutable-by-type* hole at the type surface. It does NOT prove
key-immutability across a distributed placement change (I2+, IDR-001/IDR-002 territory)
and does not unify the workspace's several per-layer ShardKey types ‚Äî each layer still
owns its own key shape by design (OWN-001).
