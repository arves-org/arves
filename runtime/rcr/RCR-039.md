# RCR-039 — Group commit / batched-fsync WAL (v1.1)

**Status:** APPLIED (2026-07-06) · **Class:** additive (new `Wal::append_group`
trait method WITH a default; `FileWal` override; new `RefKernel::commit_group`
inherent method — the frozen `Kernel` trait and the single-`commit`/`append`
paths are byte-unchanged) · **Closes:** the concrete deferred v1.1 debt the L4
report measured — the **fsync-per-commit throughput ceiling**
(`RUNTIME_FREEZE_v1.0.md`; `verification/industrial/L4_REPORT.md`).

## Motivation

The L4 industrial perf harness measured commit throughput on the real
fsync-durable `FileWal` at **~1,300 commits/s**, fsync-bound: every commit calls
`sync_all` before returning `Ok` (`FlushFileBuffers` on Windows). That is the
honest cost of the CP durability guarantee, and the L4 report named a batched /
group-commit path as the motivation for this RCR. This closes it.

## The change (assigned crates only)

**`arves-persistence`** — `Wal::append_group(Vec<PendingRecord>) -> Result<Vec<Offset>, WalError>`:
- Added to the trait **with a default** (falls back to per-record `append`, one
  sync each) so every implementor gets it without touching its `append` path.
- **`FileWal` overrides** it: write every frame, then issue **exactly ONE**
  `sync_all` for the whole group. Offsets are dense and in input order. A group
  crossing a segment-rotation boundary syncs the outgoing segment before sealing
  it, so no frame is ever left un-synced. The default per-record `append` is
  byte-identical and untouched.

**`arves-kernel`** — `RefKernel::commit_group(Vec<ProposedWrite>) -> Result<Vec<BatchOutcome>, BatchError>`:
- The **same gateway admission** as `commit`/`commit_batch` (ORCH-004
  idempotency, RCR-005 content-integrity, IDR-004 cross-shard refusal), reusing
  the `BatchOutcome`/`BatchError` types.
- Validate-first (nothing appended on any validation-class refusal) → partition
  into FRESH vs already-committed (intra-group identical duplicates collapse to
  one append, first `fresh:true`, rest `fresh:false`) → **ONE** coalesced
  `append_group` for the fresh set → **ack (insert truth) only AFTER** the
  coalesced fsync returns.

## Durability preserved (HARD RULE 4)

- **No ack-before-durable.** Truth enters the Kernel truth set only after the
  single fsync makes the whole group durable. A crash mid-group (before that
  fsync returns) leaves an *un-acked* tail: recovery truncates only the torn
  final frame (existing torn-tail detection), while any fully-written frames the
  OS already persisted survive, are recovered, and are idempotently reconciled on
  retry (ORCH-004). No acked truth is ever lost, and committed truth stays a pure
  function of inputs.
- **Deterministic committed truth.** Records are appended in input order with
  dense offsets — the identical committed content and order N sequential commits
  produce. Proven: the grouped `truth_hash` **equals** the non-grouped path (unit
  test AND the L4 perf gate), and the `FileWal` group segment bytes are
  byte-identical to N sequential appends.

## Honest atomicity boundary

As `commit_batch`: all-or-nothing covers the **validation class** (the failures a
caller can cause). A mid-group host **I/O** fault surfaces loudly as
`BatchError::PartialApply` — the reference WAL has no multi-record rollback
primitive (a Raft log entry naturally carrying a batch is I2 work).

## Proof

`runtime/crates/arves-persistence/tests/group_commit.rs` (5): durable+ordered
recovery from a fresh store; group bytes == N sequential appends; a torn group
truncates only its torn final frame (any intact un-acked frames survive, dense +
no gap); foreign-shard refused before any
write (SHARD-001); durable across a rotation boundary.

`runtime/crates/arves-kernel/tests/group_commit.rs` (8), incl. the **fsync-count**
proof via a counting `WalStore` double — a group of N does **1** sync, N separate
commits do **N**; grouped `truth_hash` == sequential; grouped commit durable on
the real `FileKernel` (drop+recover, no acked-but-lost); ORCH-004 within-group
dedupe; RCR-005 fork refuses the whole group with no append; intra-group fork and
cross-shard refused; empty group no-op.

`cargo test --workspace` **284 → 300 / 0** (this lane adds 13 group-commit tests).

## Measured evidence (honest: this host, single-node, single shard)

Re-ran `verification/industrial` (`cargo run --release --bin l4_report`), which
now measures both paths side by side and regenerates `L4_REPORT.md`. On
`windows`/`x86_64`, group size 64:

| Commits | Per-commit (c/s) | Group-commit (c/s) | Speedup |
|--------:|-----------------:|-------------------:|--------:|
| 500 | 1260 | 47983 | 38.1x |
| 1000 | 1335 | 51076 | 38.3x |
| 2000 | 1379 | 35584 | 25.8x |
| 4000 | 1306 | 33736 | 25.8x |

The `truth_hash` per load is **byte-identical** across both paths (and identical
to the previous per-commit-only `L4_REPORT.md`) — the fsync coalescing changed
only throughput, never truth. Numbers are measured on this host, not promised;
single host / single process / single shard — no network, no cluster.

## Freeze discipline

`runtime/` bytes changed under this RCR (`arves-persistence/src/lib.rs`,
`arves-kernel/src/lib.rs`, two new test files). The freeze manifest re-baseline
