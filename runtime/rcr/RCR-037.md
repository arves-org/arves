## The gap this closes (recorded by RCR-031)

`arves-conformance::live` grew a live `NodeProbe` for every pipeline node it had
exercised — Information, Kernel, Query, Control Plane, Capability, Execution,
Living Cognitive World — but **never for the `Engine` node** (`PipelineNode::Engine`,
lib.rs Part 7: "pure invocation; output is inference, NOT persisted truth"). No
live artifact therefore *composed* an Engine claim; the Engine node's conformance
lived only in `arves-engine-fabric`'s own unit tests, never in a composed
`ConformanceArtifact`. RCR-031 recorded this as the honest open gap. This RCR
closes it: **every data-plane pipeline node now has a live probe.**

## HONEST LANGUAGE (load-bearing)

- The engine driven is the **reference `PureEngine`**, **in-process** — not a
  full-fidelity cognitive engine. The probe proves the FABRIC's *contract*
  (purity, key derivation, determinism enforcement, proposals-not-commits), not
  that a real model is correct.
- The RCR-012 determinism check is a **PROBE, not a proof**: `invoke_enforced`
  double-invokes a self-declared `Deterministic` engine and compares — an engine
  whose nondeterminism is input-scoped or slower than back-to-back invocation can
  evade it (the fabric documents this same honesty boundary). The **key check is
  exact**; the determinism check is a probe.
- **ENG-001..005 stay PROPOSED and are NOT asserted.** Every Engine-node check
  derives from a **REGISTERED** invariant (ORCH-001/003/004). "ENG-003" appears
  only as the descriptive name of the proposals-not-commits behaviour, which is
  asserted *as ORCH-001* ("only the Kernel owns truth").
- What is ENFORCED vs advisory: the fabric-derived content-addressed key
  (ORCH-004) is cryptographically exact (ACS-001 SHA-256 address). The
  proposals-not-commits property is structural + behavioural (the `Engine` ABI has
  no commit path; a real `MemKernel` shows invoking the engine commits nothing
  and only `Kernel::commit` promotes the proposed effect to truth).

## Invariants derived by the Engine probe (all from behaviour, never hardcoded)

| Registered invariant | Engine-node derivation |
|---|---|
| **ORCH-004** (idempotent + content-addressable) | Engine **purity**: `invoke_enforced` on `PureEngine` twice with the same input yields the same `Inference` and the same key; the key equals the FABRIC's `invocation_key(manifest, input)` (ACS-001 address, engine cannot self-key); a different input yields a different key. |
| **ORCH-003** (replayable from trace) | **Determinism enforcement**: a self-declared `Deterministic` engine whose output actually varies is REFUSED (`FabricViolation::NondeterministicOutput`) **before any effect escapes** (`invoke_enforced` returns `Err`, no `ProposedEffect` reaches the caller) — so the recorded inference is a sound replay basis, never a silent recompute; the honest engine replays bit-identically. |
| **ORCH-001** (only the Kernel owns truth) | **Proposals-not-commits** (ENG-003): invoking the engine commits NOTHING on a real `MemKernel` (`committed_count` unchanged); the engine returns proposals only; truth appears ONLY when the Kernel commits the proposed effect. |

## Determinism

All new tests are deterministic: fixed byte inputs, no clocks, no RNG, no sleeps,
no OS entropy. `invoke_enforced`'s internal double-invoke of a `Deterministic`
engine is itself deterministic. `conformance_live` is a pure function of the
frozen runtime.
