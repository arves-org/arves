## I2 MILESTONE SUMMARY ‚Äî what the Cluster Kernel series (RCR-019..022) delivers

**Delivered (real, executable, deterministic):**

- **A per-shard Raft consensus core** (RCR-019, `arves-consensus::raft`): pure
  step-function state machine ‚Äî terms, seeded-timeout election, replication, quorum
  commit with the ¬ß5.4.2 guard, catch-up backtracking, stale-leader step-down ‚Äî with the
  four Raft safety properties re-checked after EVERY message step of EVERY test, and the
  first implementation of the frozen `ShardConsensus` contract (`SimShardConsensus`).
- **Joint-consensus membership + leadership transfer + the per-shard instance map**
  (RCR-020): IDR-003 two-phase C_old,new ‚Üí C_new under the dual-majority overlap rule
  (no-disjoint-majorities proven mid-transition), removed-server non-disruption,
  loss-free transfer, and IDR-001's "one independent Raft group per immutable ShardId"
  with proven blast-radius isolation (SHARD-001).
- **The CLUSTER KERNEL** (RCR-021, `arves-kernel::cluster`): the frozen `Kernel` trait
  over per-shard Raft ‚Äî leader-only commit (OWN-001/IDR-004, `NotLeader` live), the
  IDENTICAL RefKernel gateway admission before replication (ORCH-004 dedupe + RCR-005
  content-integrity, shared not forked), ack only after quorum (`NotReplicated` live,
  IDR-001 CP), deterministic follower apply producing byte-identical per-shard truth on
  every replica (ORCH-003 across nodes), Kernel snapshot install + crash recovery by
  local-WAL replay.
- **The adversarial distributed proof layer + the distributed conformance artifact**
  (RCR-022, this record): symmetric partitions, old-leader-returns, duplicate/reordered
  message storms, full-cluster rebuild-from-WAL ‚Äî plus S-I2-6 as a live
  `ConformanceArtifact` (`Verdict::Pass`) with the honest scope pinned inside the
  artifact.

**Design ¬ß5.2 scenario coverage (honest ledger):** S-I2-1 ‚úî (RCR-021) ¬∑ S-I2-2 ‚úî (the
apply loop replays committed outcomes; engines are never consulted on any apply/replay

**HONEST SCOPE ‚Äî what is simulated vs real (whole series):** the cluster is an
**in-process deterministic simulation** exactly as the design scopes it (¬ß3.7: the core
is a deterministic function of (messages, timers-as-events); sockets arrive only at the
step where they are the property under test ‚Äî never reached inside I2). Transport is a
FIFO bus; faults (partitions, drops, duplication, reordering) are scripted bus
behaviours; time is the injected logical tick; durability is the Arc-shared `MemWalStore`
(the same substrate/logic split I1 proved with `FileKernel`). **Real:** the Raft state
machine and its safety properties, joint consensus, the leader-only quorum-gated commit
gateway, byte-identical follower truth, exactly-once semantics under storms, and
deterministic rebuild-from-WAL. **NO network exists, NO network fault-tolerance, latency,
or multi-host operation is claimed.**

**What I3+ inherits (recorded debt, none silent):**

  replica's durable WAL receives applied outcomes; making the raft log ITSELF the

  persistence-wiring stage.

- **Read tiers I2.9 / S-I2-7** (OQ-6 ‚Üí IDR; load-bearing in I3 Distributed Query).
- **Protocol-driven snapshots + raft-log compaction** (OQ-1 ‚Üí IDR).
- **Shard placement/rebalancing** (OQ-8 ‚Üí minimal IDR; full scheduling is I4).
- **Multi-node threat model** (OQ-7 ‚Üí IDR + security review; authenticated commit stays
  v2.0 debt #8).
- **¬ß5.3 fidelity gap (recorded, not delivered):** the EXHAUSTIVE
  leader-kill-at-every-commit-path-step sweep (¬ß3.21 acked-but-lost risk row, ¬ß3.18
  zero-leakage metric) and the dup/reorder-storm **combined with** membership churn.
  Delivered tests only approximate the former (leader isolated after
