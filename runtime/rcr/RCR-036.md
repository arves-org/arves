# RCR-036 — Authenticated commit (HMAC-SHA256, trusted-key model) (Runtime v1.1)

**Status:** Applied (v1.1) · **Type:** additive, non-breaking · **Gate:** `cargo test --workspace`
must stay green (was **265**) → **277 passed / 0 failed** with this lane in isolation
(**+12**); **279** in the shared working tree alongside the co-landing RCR-037 (which adds the
other +2).

## Motivation

Freeze backlog item **#8** is the largest v1.0 trust caveat: *"`Kernel::commit` carries no
principal/authN/authZ"* — v1.0's threat model is a **trusted single host**. RCR-002 addressed the
*persistence* half (a tamper-EVIDENT SHA-256 hash-chain digest over the committed WAL trace,
`FileWal::integrity_digest`). This RCR closes the *commit-path* half for the **trusted-key model**:
a commit can now be **cryptographically authenticated** to a key-holding principal, and any
tampering of the committed record is detected — not merely evidenced after the fact, but **rejected
at the gateway** before it becomes truth.

Maintainer-directed, proactive hardening executed through the sanctioned RCR path — not a silent
edit.

## Change (what shipped)

`runtime/crates/arves-kernel` (the sole commit gateway crate):

- **`auth` module** — a dependency-free **HMAC-SHA256** (RFC 2104) built on the runtime's OWN
  SHA-256 (`arves-acs::sha256`; ZERO new third-party crates). Verified against **RFC 4231** known-
  answer vectors (cases 1, 2, 3, 6). Public helpers: `hmac_sha256`, `commit_mac` (the one
  authoritative MAC binding, used by both signer and verifier), `ct_eq` (constant-time compare),
  `genesis_anchor`/`fold_anchor` (the authenticated-commit anchor chain).
- **`Principal(String)`** — the identity of a commit authority (a shared-key holder).
- **`RefKernel::register_principal(principal, key)`** — provisions a principal's shared HMAC key
  in memory (the trusted-key model's key-distribution seam; never persisted).
- **`RefKernel::commit_authenticated(proposed, principal, mac)`** — verifies the MAC against the
  registered key (constant-time), then commits through the **IDENTICAL** sole gateway
  (`commit_inner` — so ORCH-004 idempotency and RCR-005 content-integrity are unchanged), then
  **anchors** the record. A missing key or a non-verifying MAC returns the new
  **`CommitError::AuthenticationFailed { shard, principal }`**; nothing is committed; the anchor is
  unchanged.
- **`RefKernel::authenticated_digest()`** — the authenticated-commit **anchor**: a hash-chain
  `aᵢ = SHA256(aᵢ₋₁ ‖ principal ‖ mac ‖ content ‖ offset)`, computed with the same dependency-free
  SHA-256 as RCR-002. It advances **iff** an authenticated commit lands, is principal-attributable,
  and is the companion to RCR-002's WAL `integrity_digest` (test #7 shows both chains move
  together).

**Additive & backward-compatible.** The frozen `Kernel` trait is untouched; the unauthenticated
`commit` path is byte-for-byte unchanged (v1.0 trusted-host mode still works and leaves the anchor
at genesis). One new `CommitError` variant was added (all workspace matches on `CommitError` have a
wildcard arm except the exhaustive `Display`, which gained an arm). One new **internal** dependency
edge `arves-kernel(rank 40) -> arves-acs(rank 15)` — downward-only, LAYER-001 legal (verified by the
conformance dependency-graph property test in the green run).

- `tests/authenticated_commit.rs`:
  1. `valid_mac_admits_and_advances_anchor` — a valid MAC admits; truth grows; anchor advances off
     genesis.
  2. `forged_mac_rejected` — a forged MAC → `AuthenticationFailed`; **no truth**; anchor unchanged.
  3. `tampered_payload_rejected` — a captured MAC replayed with a swapped payload is rejected (the
     MAC binds the payload bytes).
  4. `unknown_principal_rejected` — no provisioned key → rejected.
  5. `anchor_changes_iff_trail_changes` — distinct records change the anchor; the SAME trail on an

     and an idempotent re-commit leave the anchor untouched.

  7. `authenticated_commit_advances_wal_integrity_digest` — anchoring composes with RCR-002: the
     on-disk WAL `integrity_digest` also advances once an authenticated record is committed.

## Scope (honest — what this IS and is NOT)

- **IS: a real MAC.** HMAC-SHA256 over a canonical, length-prefixed binding (domain tag ‖ principal
  ‖ shard ‖ content ‖ payload). A party without the key **cannot forge** a MAC, and any change to
  any bound field changes it — so a valid MAC proves the commit came from a key-holder AND that the
  record was not tampered. Verification happens **at the gateway, before admission** (rejection, not
  just evidence).
- **IS: attribution + tamper-evidence anchored.** Each authenticated record's MAC is hash-chained
  into the Kernel-owned anchor, which composes with RCR-002's WAL digest. A forged record without
  the key cannot extend the trail undetected.
- **IS NOT: enforced across production callers — it is OPT-IN.** `commit_authenticated` is an
  *additive* inherent `RefKernel` method. The frozen `Kernel` trait is untouched, and every
  production commit path (the bridge, `ClusterKernel::commit`, and `commit_inner`) still routes
  through the byte-for-byte unchanged **unauthenticated** `commit`. This RCR ships an authenticated
  *primitive/mechanism* a deployment can adopt; it does **not** retroactively authenticate existing
  callers, so **no already-deployed trusted-host caveat is silently reduced**. What changed is that
  an authenticated commit path now *exists* alongside the still-fully-open default — the score claim
  must read "an opt-in authenticated-commit mechanism now exists", not "commits are now
  authenticated".
- **IS NOT: public-key non-repudiation.** HMAC is **symmetric / shared-key**: signer and verifier
  hold the same secret, so attribution is *repudiable* — it proves "*a* holder of this principal's
  key", not "*this principal and no one else*". True non-repudiation needs an ed25519-class
  signature = a crypto dependency = a **separate v2.0 RCR** (still open).
- **IS NOT: key management or authZ.** **Key distribution is out of scope** (the operator
  provisions shared keys via `register_principal`; the trusted-key model). No authorization/tenant-
  scope policy is implied — `Principal` carries identity, not permissions.

## Follow-ups (still open — v2.0)

- Public-key **non-repudiation** (ed25519-class signatures) — needs a scoped crypto dependency.
- Key management: distribution, rotation, revocation; authZ / tenant-scope on the authenticated
  principal.
- Durably persist + verify the authenticated anchor on recovery (auto-detect on restart, not only
  on explicit audit) — pairs with RCR-002's checkpoint-boundary follow-up.
