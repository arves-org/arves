# RCR-015 — Durable-WAL bridge flag `--wal-dir` (v1.1)

**Status:** APPLIED (2026-07-05) · **Class:** additive, backward-compatible (v1.1)
**Companion:** RCR-014/RCR-016 (shard selection + bind verb) — the multi-tenant bridge
session now also survives its process.

## The gap (product need)

The `arves-bridge` bin always constructed `MemKernel` — **truth died with the process**.
Every product built on the bridge (P4/P5 OSes, capstone) lost its entire committed world
on restart, while the runtime underneath has had a fsync-durable, crash-consistent,
deterministically-recovering `FileKernel`/`FileWalStore` since I1.5 (incl. checkpoint +
lossless-or-loud recovery, I1.6/I1.7). Only the bin withheld it: no product-side
workaround exists because products speak the line protocol and cannot choose the Kernel.

## The change (additive)

**Server** (`crates/arves-bridge/src/bin/arves-bridge.rs`):
- New CLI flag `--wal-dir <path>`: the bin constructs
  `FileKernel::try_recover(FileWalStore::open_root(path))` — every commit is
  fsync-durable and startup RECOVERS the directory's committed truth by deterministic
  replay (ORCH-003, lossless-or-loud: an unrecoverable directory refuses startup, exit 66,
  never serves partial truth).
- **No flag → `MemKernel`, byte-identical** to before (all pre-existing protocol tests
  exercise this arm unchanged).
- An unknown argument or a missing path is refused loudly (exit 64) — a mistyped
  `--waldir` silently falling back to volatile memory would be a durability trap.
- The session loop is factored into `serve<K: Kernel>` so both arms run the IDENTICAL
  protocol logic — the flag changes durability, never behaviour.

**Client** (`products/arves-sdk-ts/src/bridge.mjs` — living, product-side):
- `KernelBridge` options gain `{ walDir }` → appends `--wal-dir <walDir>` to the spawn
  args (validated non-empty string). Absent → no args added, exactly as before.

## Proof (durability end-to-end)

- 2 new server tests (`rcr015_*`): (a) session 1 commits over a wal-dir and the Kernel is
  **dropped**; session 2 `try_recover`s the same directory and the SAME body answers
  `already-committed` with the SAME ContentId **and index 0**, and a new body lands at
  index 1 (recovery is exact, ORCH-003); (b) protocol parity — the full id/invoke chain
  behaves identically over the durable Kernel. `cargo test --workspace` 106 → **108/0**.
- 2 new product regressions (`products/robustness.test.mjs`, 53 → **55/55**) over the
  REAL exe: process 1 (`--wal-dir <tmp>`) commits and is **hard-killed** (`p.kill()`,
  no clean shutdown); process 2 over the same directory answers the same body
  `already-committed` with the SAME ContentId+index — a true kill/restart durability
  round-trip; plus a client-level `{ walDir }` wiring bite (temp dir, cleaned up).

## HONEST SCOPE (stated in the bin header and the client doc)

**Single-host durability.** Integrity = per-frame CRC32 (error-detection) + the RCR-002
SHA-256 hash-chain digest (tamper-EVIDENT, not tamper-proof). There is **NO
authentication/authorization** on commit (v2.0 debt #8 in `RUNTIME_FREEZE_v1.0.md`) and
no replication — per-shard Raft is I2+ work (IDR-001..005 untouched). The recovery test
is a proof of deterministic recovery on this reference runtime, not a distributed
durability claim.
