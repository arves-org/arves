# RCR-007 — SHARD-001 two-tenant isolation test at the truth gateway (Runtime v1.1)

**Status:** Applied (v1.1) · **Type:** additive (test-only + a one-line catalog citation update)
· **Gate:** `cargo test --workspace` → **81 passed / 0 failed** (was 80; +1 isolation test).
Freeze baseline re-advanced via `freeze_check.py update`.

## Motivation (Madde-11 audit finding SHARD-001-F1)

The Madde-11 SHARD-001 audit confirmed isolation is genuinely **enforced at persistence** (a
per-shard WAL/dir + a biting `wrong_shard_append_rejected` test), but found the **Kernel** had
**no** cross-tenant negative test: every Kernel test hardcoded the single shard `t1/w1`, and the
PropertyCheck catalog cited SHARD-001 to *structure* ("commit keys truth by (tenant, workspace)")
rather than to a named biting test the way ORCH-004 does. The invariant most readers understand as
"tenants cannot see each other" had no executable "a shard MUST NOT contain cross-tenant data"
proof at the truth-owning component.

## Change (what shipped)

- New Kernel test `behaviour_8_two_tenant_isolation` (`walking_skeleton.rs`): two tenants commit
  under the **same content bytes** but **different shards** →
  - they are **distinct** truths (`committed_count == 2`; no cross-tenant dedup — the shard is part
    of identity), and
  - each shard's `snapshot_shard` carries **only** its own tenant's payload — neither tenant's
    secret appears in the other's snapshot (`contains_bytes` cross-leak assertions).
- `arves-conformance::property_check`: the SHARD-001 catalog entry now cites the **named** tests
  (`behaviour_8_two_tenant_isolation` + persistence `wrong_shard_append_rejected` /
  `multi_shard_isolation_survives_disk`) instead of a structural description — SHARD-001 moves from
  "cited-structural" to "cited-to-a-biting-test," matching ORCH-003/004.

## Scope (honest)

- **IS:** an executable cross-tenant isolation proof at the Kernel gateway; SHARD-001's catalog
  citation upgraded to a named test. Additive; no public API changed.
- **IS NOT:** it does not make the runtime `ShardKey` fields immutable-by-type (audit finding F2 —
  the fields are still `pub`; the opaque `arves-invariants::ShardKey` is still unused) — that is a
  separate, larger RCR left tracked. It does not add multi-tenant support to the products (they
  remain single-tenant, tenant hardcoded at the bridge — audit F3, a documentation/scope matter).
- **Independence unchanged: G1.**
