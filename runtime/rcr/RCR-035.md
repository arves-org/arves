# RCR-035 — Bridge `scan`: distinguish an unreadable log from a never-committed shard

**Status:** Applied (v1.1) · **Type:** corrective + additive, amends RCR-033 (one
signature change on the RCR-033-introduced `scan_shard` enumerator: `Vec<…>` →
`Result<Vec<…>, ScanFault>`; one new `ERR scan-fault` protocol response on the
`scan` line; ZERO change to the never-committed / happy paths — a fresh shard
still scans `scan 0`, an enumerable shard still streams byte-identically). No
frozen v1.0 type or trait signature touched; `scan_shard` and `ScanFault` are
RCR-033-era v1.1 bridge surface, not frozen-v1.0 API. The write path and every
other verb are byte-identical. ·
**Gate:** `cargo test --workspace` 263 → **265 passed / 0 failed** (verified
first-hand; +1 bridge lib test, +1 bridge bin test). Architecture gate green (no
new dependency edge — `store.shards()` is the already-in-scope `WalStore` method).
Freeze re-baseline (`freeze_check.py update`) LEFT TO THE MAINTAINER integration
loop per this lane's mandate; NO git operations performed.

**Milestone authority:** adversarial review of RCR-033 (products/bridge lane).
The audit found that `scan_shard` swallowed EVERY read/replay fault as an empty
result (`Err(_) => Vec::new()`), so a genuinely corrupt or compacted retained
log was indistinguishable from a never-committed shard at the scan surface — both
answered `scan 0`. A reconstruction consumer (`recoverFromWal`) would read that
as "nothing was ever committed" rather than "the log could not be read", silently
masking truth-loss as emptiness. The imperfection was documented in the
doc-comment, but the honest signal was lost on the wire. A runtime gap found
during product work is an RCR, never a product-side edit (IDR-006).

## HONEST LANGUAGE (load-bearing)

- **Emptiness and unreadability are now DIFFERENT answers.** `scan_shard` splits
  the two causes of a failed `ShardProjection::at_head`: if the store holds NO
  durable log for the shard (`!store.shards().contains`), it was legitimately
  never committed to → an empty `Ok(vec![])` (the honest fresh-shard answer); if
  a durable log EXISTS but the fold could not be reproduced (open/replay fault,
  or a compacted prefix with no query-side snapshot bootstrap — RCR-023 DR-7) →
  `Err(ScanFault)`, surfaced on the wire as `ERR scan-fault`, NEVER `scan 0`.
- **Still a read (OWN-001).** The distinguishing probe is `WalStore::shards()`, a
  read-only listing; no write method is reachable and no `Kernel::commit` path is
  opened. The Query layer stays "Writes: NOTHING".
- **Determinism (HARD RULE 4) unchanged.** `scan` commits nothing; the
  fault/empty decision is a pure function of the store's committed state
  (`shards()` membership + the deterministic fold). It cannot introduce
  nondeterminism into committed truth.
- **The SDK already surfaces it.** `products/arves-sdk-ts/src/bridge.mjs`
  `scan()` throws on ANY `ERR` line, so `ERR scan-fault` is raised as a thrown
  error, not silently read as `[]` — a probe is a probe.

## Honest scope (NOT claimed)

No new durability, replication, or repair capability: `scan_shard` DETECTS an
unreadable retained log and refuses honestly; it does not recover it. A
compacted-prefix shard is unreadable at Stage-1 because there is no query-side
snapshot bootstrap (RCR-023 DR-7) — that bootstrap remains future RCR surface.
The `ScanFault` carries the shard scope only, not a fault taxonomy (open vs
replay vs compaction) — a richer read-fault variant on `QueryError` is still the
recorded future RCR (`projection.rs` `unservable`). No authN on the read (v2.0
debt #8). The effect→skill causal edge remains journal metadata; a native
attributed-INVOKE verb is still the recorded next-RCR candidate.
