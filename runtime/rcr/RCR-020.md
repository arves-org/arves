## Design resolutions (open points; simplest option consistent with IDR-001..005 — recorded, never silently invented)

| # | Resolution |
|---|---|
| DR-1 | **Config-on-append (Raft §6):** a membership entry takes effect the moment it is appended (not committed) and rolls back on truncation (`refresh_config_from_log` rescans the log after every append/truncate). This is the standard joint-consensus semantics and the only one that keeps the no-disjoint-majorities window closed. |
| DR-2 | **Joint discipline is unbypassable:** `client_propose(EntryKind::Membership)` is refused with `MembershipRejected` — the only reconfiguration door is `change_membership`, which owns the two-phase transition. Callers name only the TARGET `Membership::Stable`; a caller-supplied `Joint` (or empty voter set) is refused. |
| DR-3 | **One reconfiguration in flight per shard** (design §3.8): `change_membership` refuses while the config is joint OR while any configuration entry is uncommitted (`config_index > commit_index`). |
| DR-4 | **Learners carried opaquely:** learner ids ride in the frozen `Membership` shape and are replicated to, but no catch-up/promotion protocol exists (the frozen `Role::Learner` semantics beyond "receives log, does not vote" are a later stage). Joining voters catch up via next-index backtracking (no snapshots — OQ-1 unchanged). |
| DR-5 | **Leader self-removal:** the leader continues replicating (without counting itself toward C_new majorities — `commit_quorum_met` intersects holders with each voter set) and steps down only after C_new commits, via a SAME-TERM step-down that PRESERVES `voted_for` (resetting it at an unchanged term could grant a second vote in one term and break Election Safety). |
| DR-6 | **Transfer error mapping (frozen error enum is frozen):** non-leader source → `NotLeader`; target that is self or not a voter → `MembershipRejected` (documented reuse: "not a usable member of the current voter configuration"); a target that cannot be caught up within the sim budget → `QuorumUnavailable` (honest unavailability). No new error variant was added. |
| DR-7 | **Disruptive-removed-server mitigation = leadership check (thesis §4.2.3), not pre-vote:** replicas in fresh contact (< `ELECTION_TIMEOUT_MIN` ticks) with a current-term leader ignore non-transfer RequestVotes entirely. Chosen over pre-vote as the smaller standard mechanism that (a) the design's §3.21 risk row explicitly anticipates and (b) leadership transfer requires anyway (the `transfer` flag bypass). Pre-vote remains available to the OQ-4 timing IDR. A REJOINING (still-replicated-to) node may still force one bounded re-election via the AppendEntries/-Reply term path — that is standard Raft and terminates; a REMOVED node cannot disrupt at all. |
| DR-8 | **Shard map = registry of bound handles:** `SimShardMap` stores one `SimShardConsensus` (local-replica-bound, client-to-replica semantics as in RCR-019 DR-5) per `ShardId`; duplicate registration PANICS (IDR-001 fixes exactly one group per shard — a second group is harness programmer error, not a runtime condition, so no frozen error is abused). Static registration only: shard→node placement/rebalancing stays OQ-8/I4 scope. |
| DR-9 | **`change_membership` returns the JOINT entry's index** — the frozen trait doc says the method "begins" the change; completion is observable via `await_commit`, `membership()` and subsequent proposals. |
| DR-10 | **Leadership transfer is stage-assigned scope BEYOND the approved design ladder:** the I2 design's ladder (§3.5 I2.1..I2.9) and scenario set (§5.2 S-I2-1..8) do not mention leadership transfer; it was named in this stage's assignment. It is grounded in IDR-004 (per-shard leader election — an *orderly* per-shard leader handover is the controlled form of the election IDR-004 mandates) and the Raft thesis §3.10 (the standard transfer mechanism), and it is NOT free-standing: the §4.2.3 leadership check that the design's §3.21 disruptive-rejoin risk row explicitly anticipates (DR-7) requires the `transfer` vote flag anyway, so the transfer path is the design-anticipated mitigation's other half. Recorded here so traceability (Non-Negotiable Rule 10) rests on this row, not silently on the stage assignment. |

## HONEST SCOPE — what is simulated vs real

- **In-process-simulated:** transport (FIFO bus), message loss and partitions (bus
  filters), time (injected logical tick). NO network exists; NO network fault-tolerance
  is claimed (design §3.7: sockets arrive at the step where they are the property under
  test).
- **Real:** the joint-consensus state machine itself — config-on-append, dual-majority
  elections AND commits, two-phase C_old,new→C_new with automatic finalization, leader
  self-removal step-down, joiner catch-up, removed-server non-disruption, leadership
  transfer — plus the per-shard group independence, all under the four safety properties
  checked after every message step.
- **Not in this stage:** durability/WAL wiring (IDR-005 unification), Kernel commit-path
  wiring (separate RCR per design §6.2), read tiers (I2.9), snapshots (OQ-1), wire format
  (OQ-3), learner promotion, dynamic shard placement (OQ-8/I4).
- **Known availability caveat (safety intact, CP posture intact):** because no no-op
  entry is appended on election (RCR-019 DR-2, forced by the frozen `EntryKind`) and
  prior-term entries commit only behind a current-term commit (Raft §5.4.2 guard), a
  leader failover MID-TRANSITION parks the reconfiguration — the new leader cannot
  finalize the inherited joint entry, and rejects further changes with
  `MembershipRejected`, until the next committed client proposal in its term. This is a
  consequence of RCR-019 DR-2; it resolves when the OQ-4 timing IDR or the WAL-wiring
  stage revisits no-op-on-election. (The joint-membership tests unblock this the same
  way a live system would: by committing a client proposal after heal.)
