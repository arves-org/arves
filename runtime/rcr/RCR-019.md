## Design resolutions (open points the design left; simplest option consistent with IDR-001..005 — recorded, never silently invented)

| # | Resolution |
|---|---|
| DR-1 | **Log indexing:** first entry at `LogIndex(1)`; index 0 is the standard Raft empty-log sentinel used only in `prev_log_index` arithmetic. Indices dense, never reused (frozen `LogIndex` contract). |
| DR-2 | **No no-op entry on election:** the frozen `EntryKind` carries only `Outcome | Membership`; adding a `NoOp` variant would change the frozen contract. Raft §5.4.2 is preserved instead by the commit-advance term guard (prior-term entries commit only once a current-term entry reaches quorum). |
| DR-3 | **Timing (OQ-4 pending its IDR):** compile-time logical-tick constants — heartbeat every 2 ticks, election timeout uniform in [10, 20) ticks — drawn from seeded SplitMix64; no pre-vote in Stage 1. The OQ-4 IDR may externalize/retune these; determinism story is already settled (seed = the entire randomness budget). |
| DR-4 | **Stage-1 log lives in memory inside the core.** No durable artifact is created, so no IDR-005 second-log drift is possible yet; WAL-as-Raft-log wiring (log = WAL = decision trace) is the persistence-wiring stage and will consume the core's log view. |
| DR-5 | **Harness shape:** single-threaded `SimCluster` + `Rc<RefCell<..>>` handles; `SimShardConsensus` uses client-to-replica semantics (a handle is bound to one local replica), which is what makes the OWN-001 follower-rejection test expressible through the frozen trait. |
| DR-6 | **Stage scoping of the frozen trait:** `change_membership` → `MembershipRejected` (joint consensus is ladder step I2.8); `read_index(Linearizable)` is leader-only and returns the leader commit index (real read-index/lease protocol is I2.9/OQ-6); weaker tiers answer with the honest local commit position. No snapshot install (OQ-1): catch-up is log backtracking only. |
| DR-7 | **Ladder resequencing:** design §3.5/§3.20 orders the ladder "replication first, election last" (I2.1→I2.7), but this stage lands I2.5+I2.6+I2.7 (including election) BEFORE the I2.1–I2.4 Kernel-truth replication steps. Rationale: I2.1–I2.4 prove Kernel-truth apply and therefore require the separate Kernel-wiring RCR the design itself mandates (§6.2 — consensus core lands first, Kernel commit-path wiring is its own RCR); the consensus core is self-contained inside `arves-consensus`, and the deterministic simulation (per-step safety checking, scripted partitions) preserves the failure-isolation intent of the ladder's ordering. The ladder's step CONTENT is unchanged — only the landing order across RCRs. |
| DR-8 | **`await_commit` identity semantics (frozen contract is index-only):** `Ok(entry)` at the proposed index does NOT guarantee it is the caller's proposal — after a failover, a deposed leader's un-replicated entry can be superseded and a DIFFERENT entry (committed by the new leader) can legally occupy that index; `await_commit` then returns that foreign entry. This is legal Raft and legal under the frozen signature (safety holds — no divergence), but callers MUST compare the returned `LogEntry`'s `Outcome` digest against their own proposal before treating the commit as theirs. The future Kernel commit-path wiring RCR (design §6.2) must perform exactly that check. Documented on `SimShardConsensus::await_commit`. |

## HONEST SCOPE — what is simulated vs real

- **In-process-simulated:** transport (FIFO bus), message loss and partitions (bus
  filters), time (injected logical tick). NO network exists; NO network fault-tolerance
  is claimed. This is exactly the vehicle the design mandates for I2.1..I2.6 (§3.7:
  sockets arrive only at the step where they are the property under test).
- **Real:** the Raft state machine itself — terms, votes, election, replication,
  quorum commit, conflict truncation, catch-up — and the four safety properties, checked
  after every single message step of every test.
- **Not in this stage:** durability/WAL wiring (IDR-005 unification), Kernel commit-path
  wiring (separate RCR per design §6.2), joint-consensus membership (I2.8), read tiers
  (I2.9), snapshots (OQ-1), wire format (OQ-3), multi-shard dispatch (the core is one
  group; per-shard fan-out arrives with the Kernel wiring).
