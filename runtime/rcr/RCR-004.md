# RCR-004 — Native Rust ACS-003/004/005 semantic validators (Runtime v1.1)

**Status:** Applied (v1.1) · **Type:** additive (new module + tests; no existing behavior/ABI
change) · **Gate:** `cargo test --workspace` must stay green → **77 passed / 0 failed** after this
change (was 75; +2 semantic tests). Freeze baseline re-advanced via
`verification/freeze/freeze_check.py update` as part of this RCR.

## Motivation

CCP-006 (Kit 0.3.0) + CCP-007 (0.3.1) shipped **19 semantic negative vectors** (7 `envelope` +
8 `instance` + 4 `language`) for the ACS-003/004/005 reject surfaces. Those vectors decode
CLEANLY as canonical dCBOR (the ACS-002 byte layer accepts them); the defect is purely semantic,
one layer up. The frozen Rust v1.0 reference had **no** ACS-003/004/005 validators, so it
**declared the semantic tiers DEFERRED** (like the `nfc` tier), and only the living Python
reference validators exercised them. That deferral was recorded as the open RCR-004 debt in
`standard/acs/CCP-GATE-Ratification-v2.md` and the evidence docs. This RCR retires it at the
source of truth: the reference runtime now rejects the semantic tiers **natively**, so a G2 party
can diff against a Rust reference for the full ACS surface, not just ACS-001/002.

Design record (design-before-code, per CLAUDE.md): `verification/rcr-drafts/RCR-004-runtime-v1.1-DESIGN.md`.

## Change (what shipped)

- **New module `arves-conformance::semantic`** (`runtime/crates/arves-conformance/src/semantic.rs`):
  `validate_envelope` (ACS-003 §5/§6.3), `validate_instance` (ACS-004 §6.5/§7/§8, against a decoded
  schema document), and `check_term_set` (ACS-005 §8/§9.2/§11), plus a `SemanticReject` enum whose
  `.code()` returns the **registered CCP-006 kebab reason code** (`missing-required-field`,
  `unknown-field`, `field-type-mismatch`, `value-out-of-range`, `malformed-content-id`,
  `empty-shard-scope`, `cardinality-violation`, `provenance-invariant`, `terms-not-sorted`,
  `duplicate-term`, `malformed-term-list`). Re-derived from the ACS-003/004/005 spec text; mirrors
  the from-scratch Python reference validators (which emit prose — the Rust arm emits codes).
- **`arves-conformance::lib`**: `pub mod semantic;` added (one line).
- **Tests (the acceptance proof):** `semantic_rejects_frozen_vectors` reads the frozen
  `standard/vectors/acs_negative_vectors.tsv`, decodes each `envelope`/`instance`/`language` row,
  runs the matching validator, and asserts it is **REJECTED with the exact registered code** —
  **7 + 8 + 4 = 19/19**. `semantic_accepts_golden_positives` asserts the golden instance + a valid
  GL term-set are ACCEPTED (no false rejects).

## Scope (honest — what this is and is NOT)

- **IS:** the reference runtime's native ACS-003/004/005 semantic reject surface, byte-parity with
  the 19 frozen vectors, proven by an in-crate test that reads the shipped Kit vectors. Additive:
  no existing public trait/type/signature changed; the ACS-002 encoder/decoder and the golden/
  negative generators are byte-identical; the 75 prior tests are unchanged.
- **IS NOT (deferred to RCR-004b):** *harness exposure.* The certification harness
  (`certify_runtime.py` / `verify_runtime_sound.py`) and the `conformance` runner do **not** yet
  drive the Rust semantic validators (no `acs_validate` line-protocol bin / runner display yet), so
  a certification run still reports the Rust arm's semantic tiers as deferred. The capability now
  EXISTS and is proven in-crate; wiring it into the inputs-only grader is a small follow-up.
- **Independence unchanged: still G1.** These validators were authored in-program; making the Rust
  reference reject the vectors raises self-conformance — it does **not** manufacture G2.
