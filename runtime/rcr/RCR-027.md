## Design resolutions (open points; simplest option consistent with IDR-001..005 — recorded, never silently invented)

| # | Resolution |
|---|---|
| DR-1 | **OQ-2 (code home) → option (a), Control-Plane half:** scheduling behaviour lands ADDITIVELY beside the contract-only `arves-control-plane` contracts (RCR-026 DR-1 landed the fabric half in `arves-capability-fabric`). Per the design's Terminology note the "scheduler" is a composite of three frozen-spec roles — Capability Planner + Execution Planner (Control Plane, Vol 9 Part 4) + the mechanical dispatch runtime (Data Plane, Vol 9 Part 12) — hosted as ONE module because the DECISIONS are Control-Plane-owned; no new crate, no new layer rank, no frozen-contract change. |
| DR-2 | **Placement policy (design §6.2 routes the algorithm to IDR-007):** Stage-2 adopts a REFERENCE policy — **shard-leader affinity** for commit-bearing invocations (their proposed writes commit through that leader anyway, IDR-001; affinity saves a routing hop) and a **seeded deterministic FNV spread over healthy nodes** for `Pure` invocations (IDR-001 compute-anywhere). Recorded as explicitly NON-NORMATIVE pending IDR-007; conformance asserts behavioural properties, never the placement choice itself (design §1.9). |
| DR-3 | **Backpressure semantics (IDR-007-gated per design §6):** simplest bounded admission — a per-shard queue-depth bound; overflow is a VISIBLE, retriable `AdmissionDenied` (never a silent drop, never a global limiter — SHARD-001 tenant isolation is the scoping). Failure-Policy-driven degrade/escalate responses (Engine Graph Part 3) are NOT modeled (the manifest subset has no Failure Policy field — recorded v1.1 manifest debt); deny is the only Stage-2 response. Reference semantics, non-normative pending IDR-007. |
| DR-4 | **Retry discipline:** retriable = `NotLeader` / `NotReplicated` only (IDR-004 leadership/quorum loss; CP posture "unavailable > divergent"). Deterministic refusals — gate denials, `ContentIntegrity` (RCR-005), `Rejected`, commit-time `UnknownShard` — quarantine IMMEDIATELY (retrying a pure function of the same state reproduces the refusal). DEFERRAL (no reachable leader / no healthy compute node) is not a retry and spends no budget (the shard is lawfully unavailable; other shards keep scheduling — SHARD-001). Budget exhaustion → quarantine (F-POISON at the commit seam), never a wedged queue. **REVISED (adversarial pass) — quarantine classes:** budget-exhausted quarantine is RETRIABLE-class (`WorkState::Quarantined { retriable: true }`): a later RE-SUBMISSION of the same invocation RE-ADMITS it with a fresh budget at the dedupe step instead of collapsing onto the refusal, so a transient outage never becomes a permanent refusal inside a long-lived scheduler (proven: re-admission test; honest note — the recorded inference was discarded at quarantine, so re-admission RECOMPUTES: lawful at-least-once compute). Deterministic refusals stay terminal-class (`retriable: false`): re-running a pure function of the same state reproduces the refusal, and dedupe lawfully collapses later duplicates onto it (callers can distinguish collapse-onto-live-work from collapse-onto-refusal via `state_of`). |
| DR-5 | **Per-shard FIFO ordering:** the design's within-shard ordering is the plan DAG's (§3.8), but Stage-2 specs are single invocations with no edges, so the simplest deterministic discipline is per-shard FIFO; a blocked head re-queues at the FRONT and ends that shard's tick; quarantine REMOVES poison heads so FIFO can never wedge (proven). DAG-edge ordering belongs to the plan-graph/Orchestrator stage (I5), not invented here. |
| DR-6 | **Replay-from-record retries (ORCH-003):** the gated `Inference` is recorded in the ephemeral ledger under its ORCH-004 key at compute time; a RETRY re-proposes the RECORDED effects and never re-invokes the engine (proven: leader-loss test, engine count stays 1). This also closes the nondeterministic-recompute hazard (design §3.16: replay prefers record over recompute — a recompute could propose different content and fork intent; the record cannot). Across scheduler DISCARD the record is gone: recompute is lawful (at-least-once compute), and truth converges at the Kernel (ORCH-004) **provided the engine actually reproduces its proposed effects byte-identically** — a misdeclared engine that varies across a crash-rebuild would propose different content and add truths BESIDE any pre-crash partial commits (a fork of intent the runtime cannot detect at this seam; design §3.1.4 caveat, v1.1 debt #2). The crash-rebuild proof uses a truly reproducible engine; convergence-after-discard is claimed ONLY under that declared-honesty condition — the design's "at-least-once dispatch, at-most-once truth" holds for honest engines (REVISED per the adversarial pass; the original sentence overclaimed unconditional convergence). |
| DR-7 | **Submit-time authorization-lite:** submission resolves the binding (F-UNBOUND is a hard deny BEFORE queueing), requires the bound provider hosted, and derives the fabric ORCH-004 key so dedupe + admission are total functions. The FULL Stage-1 gate re-authorizes at dispatch with a fresh `resolve` — a rebind/revoke between submit and dispatch lawfully bites the dispatch (RCR-026 DR-10), and the gate pins the version that actually runs. |
| DR-8 | **Multi-effect commits are effect-by-effect** (the exercised G1 `kit.mjs` pattern: effects committed one-by-one through the Bridge), each under `arves-acs domain::COMMIT_CONTENT` (the bridge/live-conformance convention). NO cross-effect atomicity is claimed (the cluster batch form is the deferred RCR-021 item / v1.1 debt #3); per-effect progress is recorded so a resumed commit continues where it stopped, and a re-proposed already-committed effect dedupes at the Kernel. |
| DR-9 | **Engine hosting:** one `EngineHost` table stands for the artifact set EVERY node carries (Engine Graph Part 9: scheduling needs only the manifest; artifacts are content-addressed and identical everywhere). Placement therefore assigns a deterministic node LABEL and compute runs in-process on that node's behalf — no remote execution exists or is claimed (honest scope). |
| DR-10 | **Node presence is AP input** (IDR-001 CP/AP table: "Presence / capability statistics — AP"): the `down` set is a scripted, possibly-stale observation; it steers placement/deferral but NEVER gates a commit — a stale entry costs at most a deferral or a retriable `NotLeader`, converging by ORCH-004. |
| DR-11 | **ORCH-001/ORCH-002 catalog flip (explicit, recorded):** the design §4 table makes executable ORCH-001 ("scheduler never commits truth of its own; all truth transitions carry Kernel commit provenance") and ORCH-002 ("crash-restart with identical committed truth") proofs I4 BUILD OBLIGATIONS. They are discharged by this stage's tests, so the PropertyCheck rows flip `Pending` → `CitedTest` and the suite guard moves 5→7 — with scope stated in the citation: proven at the I4 scheduling surface; the I5 Orchestrator (plan-graph execution) remains contract-only and is NOT pre-certified. The stale "contract-only, no proof yet" reason would otherwise now be false. |
| DR-12 | **Decision log ≠ decision trace:** every scheduling decision is recorded in an in-memory, discardable log (the determinism/audit instrument of this stage). Durable WAL-trace emission of scheduling decisions — and its commit granularity under F-OVERLOAD — remains design OQ-10, routed to IDR-007 / v1.1 debt #3; NOT built, NOT claimed. |
| DR-13 | **Dedupe identity scoping (adversarial pass; SHARD-001 + design §3.6):** the scheduler's work-ledger identity is the RCR-012 fabric key (manifest identity + canonical input) QUALIFIED by the capability id (length-prefixed, injective encoding — `scheduler_key`) and structurally PARTITIONED by the immutable `ShardId` (`BTreeMap<(ShardId, IdempotencyKey), WorkState>`). The bare fabric key carries no shard and no capability, so the original single global ledger would (a) let one tenant's record silently satisfy another shard's identical submission — cross-tenant work suppression, since Kernel dedupe is shard-scoped via `ProposedWrite.shard` — and (b) collapse two capabilities sharing one provider onto one work item, skipping the second one's policy verdict. Both failure modes are now refuted by negative tests (two-shards / two-capabilities). The KERNEL's truth-level identity remains pure per-shard content addressing; the frozen `InvocationKey` contract in `lib.rs` is byte-unchanged — the qualification exists only at the scheduling surface. |

## HONEST SCOPE — what this is and is not

- **Is:** deterministic cluster-wide capability scheduling over the I2/I3 in-process
  substrate — leader-affine/compute-anywhere placement, per-shard bounded admission
  (backpressure) with proven tenant isolation, quarantine-based failure isolation
  (deterministic refusals terminal; budget-exhausted quarantine re-admittable —
  DR-4), idempotent dispatch under the fabric-derived ORCH-004 key —
  shard-partitioned and capability-qualified at this surface (DR-13) — with
  replay-from-record retries, and the full chain scheduled-invocation → gate →
  engine → proposed effects → leader commit → replicated truth.
- **Is NOT (nothing beyond this is claimed):** no network/remote execution (in-process
  `ClusterSim`; placement is a recorded label — DR-9); no exactly-once COMPUTE
  (at-least-once compute / at-most-once truth is the model, design §6.1; proven
  honestly); no plan-DAG ordering, arbitration or dynamic expansion (Orchestrator, I5);
  no sagas/cross-shard flows (OQ-4); no HITL/approval sequencing (`ApprovalRequired`
  BLOCKS, RCR-026 DR-5); no distributed cancellation (OQ-6; timeouts unmodeled — the
  manifest subset carries no Timeout field, recorded debt); no durable decision-trace
  emission (OQ-10, DR-12); no Failure-Policy degrade/escalate (DR-3); no cluster batch
  commit (v1.1 debt #3); no authN/authZ (v1.0 trusted single host); external-effect
  idempotency stays a provider declaration (v1.1 debt #2). The RCR-012 determinism probe
  remains a probe. **CAP-001..009 remain PROPOSED** — nothing here enforces them;
  registered bases used throughout (ORCH-001..004, OWN-001, SHARD-001, LAYER-001,
  IDR-001..005).

## Invariants (registered only)

- **ORCH-001** — the scheduler owns no truth: its only write path is routing
  `ProposedWrite`s through the shard leader's frozen `Kernel::commit` gateway
  (structural — no other surface exists); `Pure` work commits nothing; dropping the
  scheduler loses zero committed truth (proven; catalog row now cites it, DR-11).
- **ORCH-002** — queues/ledger/decisions are ephemeral plan bookkeeping: discarded
  mid-run and rebuilt from the plan with an IDENTICAL final committed truth set
  (proven crash-rebuild; placements are serializable, discardable decisions).
- **ORCH-003** — retries replay from the RECORDED inference, never recomputation
  (proven: engine count stays 1 across a leader-loss retry); placement/decisions are
  recorded values; durable trace emission stays OQ-10 (honest).
- **ORCH-004** — the fabric-derived content-addressable key (RCR-012) is the unit of
  identity end-to-end — at the SCHEDULING surface qualified by capability id and
  partitioned by shard (DR-13), at the KERNEL as pure per-shard content addressing:
  submission dedupe, racing-scheduler convergence to exactly one committed truth,
  idempotent re-commits after crash-rebuild (all proven).
- **SHARD-001** — queues, admission bounds, the work LEDGER, placement and commits
  scoped by the immutable shard key; the flood-isolation test proves the other
  tenant's transcript is bit-identical and its truth byte-equal, and the DR-13
  negative test proves one shard's record can never silently satisfy (suppress)
  another shard's identical submission.
- **OWN-001** — bindings stay the registry's, truth stays the Kernel's, engines stay
  the host's; the scheduler owns only its discardable schedule (one owner per state).
- **LAYER-001** — five new edges, ALL downward from control-plane(90); ranks checked
  in `property_check.rs` before adding; executable architecture gate green over the
  full workspace.
