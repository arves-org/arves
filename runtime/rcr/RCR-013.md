# RCR-013 — Same-shard atomic batch commit (v1.1)

**Status:** APPLIED (2026-07-05) · **Class:** additive (new `RefKernel::commit_batch`
inherent method + `BatchOutcome`/`BatchError` types; the frozen `Kernel` trait is
untouched) · **Closes:** `RUNTIME_FREEZE_v1.0.md` v1.1 backlog item **3** — *"Kernel
batch-commit — atomic multi-effect ... commit."*

## Scope ruling (IDR-004)

The backlog text said "atomic multi-effect / **multi-shard** commit". The frozen
`Kernel` trait contract explicitly rules the second half out: *"There is **no
cross-shard atomic commit** (IDR-004) — multi-shard intent is a saga."* This RCR
therefore delivers **same-shard** atomic batch and **refuses** a mixed-shard batch
up front (`BatchError::CrossShard`) instead of pretending — the constitution's
instrument for cross-shard atomicity remains the saga (I2+, per-shard Raft).

## The change

`arves-kernel`:

- `commit` refactored to a shared `commit_inner` so single-commit and batch run the
  **identical** gateway logic under one state lock — batch is not a second, divergent
  commit path.
- **`RefKernel::commit_batch(Vec<ProposedWrite>) -> Result<Vec<BatchOutcome>, BatchError>`**
  under a single state lock (no interleaving):
  1. **Validate first, mutate nothing:** same-shard check (IDR-004); no fork against
     committed truth (RCR-005 content-integrity); no intra-batch fork (one address,
     two payloads). Any violation refuses the WHOLE batch — zero applied.
  2. **Apply through `commit_inner`:** fresh truths committed; identical duplicates
     (pre-committed or intra-batch) resolve idempotently to `fresh: false`
     (ORCH-004), never a fork.

## Honest atomicity boundary

All-or-nothing covers the **entire validation class** — every failure a caller can
cause. A mid-apply WAL **I/O** failure (host-level) leaves the already-appended prefix
as durable truth and is surfaced loudly as `BatchError::PartialApply { applied, index,
cause }` — the reference WAL has no multi-record transaction primitive to roll a
prefix back, and this RCR does not pretend it does. Making the apply phase itself
transactional is consensus-era work (I2: a Raft log entry naturally carries a batch);
tracked forward in the I2 design package (`docs/design/I2_Cluster_Kernel_Design.md`).

## Proof

`behaviour_9_batch_commit_atomic_validation` (walking-skeleton behaviour suite):
(a) clean batch commits all, ordered+fresh; (b) re-batch idempotent (`fresh:false`,
same truths, no duplicates); (c) a batch whose LAST entry forks committed truth
commits NOTHING (all-or-nothing bites); (d) intra-batch fork refused, nothing
applied; (e) intra-batch identical duplicate resolves to one truth; (f) cross-shard
refused (IDR-004); (g) empty batch is a lawful no-op.
`cargo test --workspace` **97 → 98 / 0**.
