## Executable proofs (all deterministic)

| Mandate item (design §4/§5.2 obligation) | Test |
|---|---|
| **(a) Torn-read impossibility** — a query NEVER observes a partially-applied RCR-013 batch: at every reader-reachable observation point the WAL head is a batch boundary; each batch is visible all-or-none on every tier; every served `observed_at` is provably a batch boundary; a REFUSED batch (intra-batch fork / fork against committed truth) leaves the projection bit-identical; pinned rebuilds at every boundary are batch-atomic; the internal `at_version` replay facility CAN pin mid-batch (demonstrated) — audit surface, never a `Query`-served position | `torn_read_impossibility_batches_all_or_none_on_every_reachable_observation` |
| **(b) Replay equivalence** — on EVERY replica: rebuilt-from-own-WAL projection == live-served read (position, bytes); two rebuilds equal; rebuilds equal ACROSS replicas (`PartialEq` + digest); full-cluster crash/recover changes nothing; every served read reproducible by a pinned rebuild at its `observed_at` (ORCH-003; §3.11/§3.19) | `replay_equivalence_rebuilt_from_wal_equals_live_projection_on_every_replica` |
| **(c) Partition reads** — 5-node, 2/3 minority partition: every minority follower keeps serving the AP tier BIT-IDENTICAL to its pre-partition capture (labeled `Eventual`, old `observed_at`), fabricates NOTHING (majority-only truth `NotFound`/absent in every read form; visible universe exactly the old prefix), strong tiers refuse; heal converges ALL FIVE replicas' projections to equality and every tier serves the new truth | `minority_partition_reads_labeled_stale_never_fabricated_and_heal_converges` |
| **(d) Determinism under message storms** — duplicate/reordered consensus delivery ACTIVE on both shard buses (counters prove the storm bit, both arms), interleaved commits THROUGH the storm: two independent identically-scripted runs produce BIT-IDENTICAL query transcripts (mid-storm AP reads AND converged strong reads, gathers, per-replica fold digests); replicas converge to identical folds within each run | `query_results_deterministic_and_replicas_converge_under_message_storms` |
| **(e) Live distributed conformance** — the Enterprise-Knowledge-Query-under-distribution artifact is `Verdict::Pass` with all required invariants/properties derived Held from real behaviour; honest fingerprint ("no network transport"); the pre-existing L1 single-node artifact stays green untouched | `live::tests::live_distributed_query_scenario_passes` (+ all prior live tests unchanged) |

## Design resolutions (open points; simplest option consistent with IDR-001..005 — recorded, never silently invented)

| # | Resolution |
|---|---|
| DR-1 | **Torn-read proof scope (RCR-013 lineage):** the batch-atomicity claim is proven at the `Query` surface for every observation point a reader can reach in this deterministic harness — `commit_batch` appends the whole batch under one state lock before control returns, so served folds only ever target batch-boundary heads (asserted: every served `observed_at` ∈ boundaries). Honest limits stated in-test: the trace itself is per-record, so `ShardProjection::at_version` CAN pin mid-batch (that is the audit/replay facility, demonstrated deliberately); `BatchError::PartialApply` (mid-batch host I/O failure) is unreachable with `MemWalStore` and remains RCR-013's honest atomicity boundary; the CLUSTER batch form (a Raft entry carrying a batch) is still the deferred RCR-021 item — no distributed torn-read claim is made because no distributed batch exists. |
| DR-2 | **"Promoted impl" in conformance = a NEW scenario, not an edit to RCR-010:** the design lists the RCR-010 `QueryProjection`/`QueryProbe` as "reused, unmodified" (§2), so the live Query node rides the promoted `arves-query` implementation via the NEW `enterprise-knowledge-query-distributed` scenario, while the existing `l1-information-kernel-query` artifact (and its probe) stays byte-unmodified and green. Both the single-node reference semantics and the distributed implementation are now conformance-observed. |
| DR-3 | **Scenario axes:** design §5.1 names axes 1+8+9 (+12 mandatory). Axis 9 (Multi-agent) participates in the design *only as concurrent-reader load* — which does not exist in the in-process deterministic harness — so the live scenario declares axes **1+8+12** and says why; claiming axis 9 with zero concurrent readers would be dishonest. Multi-agent semantics are I5 (design §6.1 #10). Same honesty for axis 8 (HighVolumeStreaming — "throughput, backpressure, tenant isolation at scale"): the live workload is a handful of commits, so axis 8 participates via its **tenant-isolation clause only** — no volume, throughput or backpressure is exercised in-process (no performance claim, §6.1 #8); the scenario comment in `live.rs` states this. |
| DR-4 | **Post-heal strong-tier availability (RCR-024 DR-8 interaction):** a heal-triggered re-election can seat a leader with NO committed entry of its current term (RCR-019 DR-2 appends no election no-op), so Linearizable/Bounded stay honestly REFUSED until one commit lands in the new term. The partition test therefore commits a post-heal marker truth before asserting strong-tier convergence — the operationally correct answer under the design (refusal is the guarantee; a current-term commit validates the read-index), never a weakening of the refusal. |
| DR-5 | **PropertyCheck extension is citation-only (RCR-006 pattern):** the catalog's proven/pending counts are UNCHANGED (5/2). ORCH-001/ORCH-002 remain `Pending` — their owning crate (`arves-control-plane`) is still contract-only, and the I3 read fabric observing committed-truth-only is *evidence toward* ORCH-001, not an executable proof of the Control-Plane obligation. No silent Pending→Proven flip. |
| DR-6 | **Storm-determinism instrument:** a textual transcript (Debug of read results, gathers, per-replica fold digests, mangle counters) compared bit-for-bit across two identically-scripted runs. Sound because every structure on the read path is deterministic (`BTreeMap` everywhere, no HashMap iteration, no clocks); counters are asserted nonzero so the storm provably bit (a no-op storm can't pass). |
| DR-7 | **"Membership change under load (IDR-003)" — NOT discharged at the query layer (recorded vehicle limitation):** design §5.2 Stage 3 requires it and §5.3 lists "joint-consensus membership change" under Distributed-tests-PASS, but the I2 kernel-layer vehicle (`arves-kernel::cluster::ClusterSim`) exposes **no membership-change surface** — only `partition`/`heal`/`crash_recover`; the joint-consensus core exists one layer down (raft `SimCluster::change_membership`, RCR-020) and is not wired up to the cluster kernel. Wiring it up is an additive I2-substrate extension outside this RCR's assigned crates and needs its own recorded DR; inventing it silently here would violate the design-is-law rule. Resolution: IDR-003 coverage for I3 **maps to the I2 raft-layer joint-membership proofs** (`arves-consensus/tests/joint_membership.rs`: add-node joint→stable convergence · mid-stream follower removal keeps committing · leader self-removal steps down and the group recovers · no-two-disjoint-majorities during transition · second change rejected while in flight) — consensus-layer evidence, NOT a query-layer read. **No query read across a membership transition is proven in I3**, and the §3.6 stale-routing safety argument (`UnknownShard` on a re-homed shard → directory refresh → retry) remains an unexecuted design argument under an actual membership transition. "Query reads across a joint-consensus membership change" is added to the I4+ inheritance list, and the §5.2/§5.3 discharge claims in this record are scoped accordingly. |
| DR-8 | **Item-by-item mapping of the remaining §5.2 Stage-3 items (approximated forms, stated exactly):** "leader kill (IDR-004)" is discharged in approximated form, not by killing a lone live leader mid-serving — leadership LOSS by `deposed_minority_leader_refuses_linearizable_but_serves_labeled_eventual` (stage-2: the partition deposes the leader; strong tiers refuse, AP serves labeled) plus the heal-side re-election/deposition path of `minority_partition_reads_labeled_stale_never_fabricated_and_heal_converges` (with the fresh-leader refusal guard `new_leader_without_current_term_commit_refuses_strong_reads_until_it_commits`, DR-4), and process DEATH by the full-cluster `crash_recover` inside `replay_equivalence_rebuilt_from_wal_equals_live_projection_on_every_replica`. "Crash-rebuild during serving" is necessarily the **sequential crash-then-serve** form in this single-threaded deterministic harness (crash/recover, then reads + rebuild equality) — no reader races a rebuild, per HONEST SCOPE ("no concurrent readers racing appliers"). |

## THE I3 MILESTONE RECORD (what I3 delivers — RCR-023 + RCR-024 + RCR-025)

**Delivered (workspace 155 → 181, all stages green):**

1. **The frozen `arves-query` contract is no longer contract-only.** `WalQuery`
   (single-node core, RCR-023) and `ClusterQuery` (distributed reads, RCR-024) implement
   the frozen `Query` trait behind byte-unchanged v1.0 signatures; `ShardProjection` is
   the deterministic per-shard WAL fold (`Proj(shard,v) = fold(apply, ∅, WAL[0..v))`).
2. **Query routing (Baseline Part 5):** canonical-text + typed-`ShardId` shard resolution
   against the I2 cluster directory (SHARD-001), with `MalformedScope` before any I/O.
3. **The IDR-001 read-tier ladder, served honestly:** Linearizable = in-process
   read-index with the Raft §6.4 current-term-commit precondition (refuses under
   partition / deposed leaders / fresh elections); BoundedStaleness = provably-zero-lag
   only, else refused with the `LAG_UNATTESTABLE` sentinel (OQ-2 IDR pending);
   Eventual = always-available AP reads, staleness LABELED. IDR-005's CP-truth/
   AP-observability split is enforced at read time and adversarially proven.
4. **Additive surfaces per design §6.2** (frozen enums untouched): `gather_read`/
   `GatheredRead` (tenant-internal non-atomic union, per-shard version vector, fail-whole
   — OQ-3 yes / OQ-4 fail-whole), `read_at_least`/`floor_of`/`FloorReadError`
   (read-your-writes floor — OQ-5 carrier).
5. **The design's §4 proof rows, adversarial form (this RCR):** torn-read impossibility
   (RCR-013 batches atomic at the Query surface), replay equivalence on every replica,
   minority-partition read honesty with zero fabrication + heal convergence, and
   bit-identical query behaviour under duplicate/reorder message storms.
6. **Conformance (design §5):** the frozen framework's "Enterprise Knowledge Query"
   reference scenario runs LIVE under distribution
   (`enterprise-knowledge-query-distributed`, `Verdict::Pass`), alongside the untouched
   single-node L1 artifact; the PropertyCheck catalog cites every I3 proof (SHARD-001 /
   ORCH-003 / ORCH-004 rows), coverage honestly 5 proven / 2 pending.

**HONEST SCOPE — what I3 is NOT (nothing beyond this is claimed):**

- **In-process simulation only** (the I2 `ClusterSim` vehicle): no network, no wire
  format, no sockets, no concurrent readers racing appliers, no real read-index
  heartbeat round (the sim-omniscient directory closes the stale-leader hazard; the
  §6.4 precondition is enforced by refusal), no network fault-tolerance claim. The
  live artifact's fingerprint says so ("L3(scoped)", "no network transport").
- **No query read across a membership change** (DR-7) — the kernel-layer
  `ClusterSim` exposes no membership API (partition/heal/crash only), so the §5.2
  Stage-3 "membership change under load (IDR-003)" item is NOT discharged at the
  query layer; IDR-003 evidence for I3 is the I2 raft-layer joint-membership suite
  only, and the §3.6 stale-routing safety story (`UnknownShard` on a re-homed
  shard → directory refresh) is unproven under a real membership transition.
- **No real millisecond staleness attestation** — the only provable lag is zero; the
  time↔index mechanism is OQ-2 and REQUIRES its own IDR before any bounded-read claim.
- **No LCW read views** (OQ-8 — semantics undefined in the frozen corpus; no edge to
  `arves-lcw` exists), **no authN/authZ** (OQ-1 — trusted single host, as at v1.0),
  **no query language/planner** (§6.1 #5), **no cross-shard atomic or
  snapshot-consistent reads** (§6.1 #3), **no Kernel read API** (§6.1 #2 — reads are
  WAL replay only), **no distributed batch** (cluster form of RCR-013 deferred),
  **no performance numbers** (§6.1 #8).
- **QUERY-001 remains PROPOSED** — cited only as proposed throughout; the read-only
  obligation is enforced via the registered A-003 Layer-Matrix row, the frozen trait
  shape, and structure (immutable sim borrows; no path to `Kernel::commit`). The §5.4
  CCP-GATE ratification scenario now EXISTS as a live artifact, but ratification is a
  maintainer-gated CCP, not this RCR.

**What I4 (Capability Scheduling) and later inherit from I3:**

- OQ-2 staleness-attestation IDR (real ms bounds for follower reads).
- Networked read-index (leadership-confirmation heartbeat round) with the transport /
  wire-format work inherited from I2 (OQ-3, RCR-021 DR-14).
- Protocol-driven query-replica snapshot bootstrap (RCR-023 DR-7: compacted-prefix
  shards are refused as unservable until then) + raft-log compaction (I2 OQ-1).
- LCW read-view semantics (OQ-8) — engineering decision, possibly IDR-grade, plus a
  CCP-GATE scenario if new guarantees are promised.
- AuthN/authZ on the read path (OQ-1; joins the RCR-#8 v2.0 authenticated-commit debt).
- Typed contract shard key or `/`-rejecting `ShardKey::new` (RCR-023 DR-2 / RCR-024
  DR-9 residue: the frozen text form stays ambiguous for `/`-bearing tenants at the
  frozen entry points).
- Session-monotonicity carrier beyond the floor (OQ-5's stated-guarantee half).
- Distributed batch commit (a Raft entry carrying an RCR-013 batch) and its torn-read
  proof in replicated form.
- **Query reads across a joint-consensus membership change** (the §5.2 Stage-3
  item NOT discharged in I3 — DR-7): an additive `ClusterSim` membership surface
  wired to the I2 joint-consensus core (`SimCluster::change_membership`, RCR-020),
  plus the §3.6 stale-routing proof (`UnknownShard` on a re-homed shard →
  directory refresh → retry) under an actual membership transition.
- QUERY-001 CCP ratification using the now-live isolation scenario (design §5.4).
