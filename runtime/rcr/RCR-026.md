## Design resolutions (open points; simplest option consistent with IDR-001..005 — recorded, never silently invented)

| # | Resolution |
|---|---|
| DR-1 | **OQ-2 (code home) → option (a):** scheduling-era fabric behaviour lands ADDITIVELY beside the contract-only `arves-capability-fabric` contracts (this lane's assigned crate), via RCR — no new crate, no frozen-contract modification. Later I4 stages (Planner/scheduler behaviour) may still choose (b) for Control-Plane-owned code; nothing here prejudges that. |
| DR-2 | **"First rung" scope cut:** the I4 design has no explicit stage ladder (unlike I3 §5.2). Stage 1 = the fabric-core subset: §3.5 lifecycle steps 1/3/10 (declare/bind/supersede/retire + authorize) and §3.1.1 (resolve per shard) / §3.1.2 (authorization gate) / §3.1.4 key-enforcement (via RCR-012), single-node in-process. Explicitly NOT built: selection policy & placement (§3.1.3 — Control Plane; placement algorithm is IDR-007-gated, §6.2 row 2, R8), admission control/backpressure (§3.1.5 — IDR-007-gated), trace emission (§3.1.7 — granularity is OQ-10), sagas (OQ-4), distributed anything. |
| DR-3 | **Revocation semantics:** the frozen contract has no unbind/revoke and the design retires bindings only by supersession ("superseded by version bump, never mutated", §3.5 step 10). Simplest consistent option: `revoke` = a TOMBSTONE event appended at a strictly-higher version (IDR-005 append-only discipline applied to configuration); resolve → hard `Unbound`; rebind must strictly supersede the tombstone, so replaying an old version can never resurrect a revoked capability. Errors reuse the frozen `RegistryError` variants (adding a variant would change the frozen enum): undeclared → `UndeclaredCapability`, nothing-active → `Unbound`, non-superseding version → `NonMonotonicVersion`. |
| DR-4 | **Pinned replay resolution:** `resolve_pinned` serves superseded and revoked-era `Bound` versions FOREVER (replay must use the binding that actually ran — ORCH-003 registered basis; Engine Graph Part 11 old-traces-stay-replayable; CAP-009 states it directly but is (PROPOSED — CCP-GATE required)); it is a pure read and NEVER an authorization surface — only `resolve` answers "what may run NOW". A tombstone version is not a binding (`Unbound`). |
| DR-5 | **Policy enforced-not-owned (design §3.1.2(b), §3.2 input row):** no policy engine exists in the runtime and policy is OWNED by Vol 17/Vol 14. The gate therefore consumes a caller-supplied `PolicyVerdict` (allow/deny/approval-required) and ENFORCES it. `ApprovalRequired` BLOCKS in Stage 1 — there is no HITL/approval sequencing surface (that is Control-Plane machinery, a later stage); refusal, never a silent allow (the axis-7 gate-must-BLOCK property). |
| DR-6 | **Gate home vs "the fabric does not perform the invocation itself":** the frozen registry doc's language governs the `CapabilityRegistry` surface, which is byte-unchanged and still never invokes/selects/commits/plans. The `gate` module is a STATELESS validation + composition surface in the same crate: it owns no state (OWN-001 — the caller's registry owns bindings), selects nothing (the capability id is the caller's plan input — selection stays the Control Plane's, Vol 9 Part 3; no "smart fabric", design §6.1), commits nothing (NO kernel dependency exists in this crate — structurally impossible, ORCH-001), and delegates the invocation DOWNWARD to `arves-engine-fabric::invoke_enforced` (LAYER-001). The crate-level "std-only skeleton" wording is honestly superseded in the header, not silently deleted. |
| DR-7 | **The bridge is UNTOUCHED:** `arves-bridge::invoke` keeps its inline gate (resolve + provider-identity + `invoke_enforced` + undeclared-effect check). This RCR formalizes the same semantics fabric-side (`engine_provider_id` is the single fabric definition of the `engine:{name}@{version}` scheme). Unifying the bridge onto `gate::invoke_gated` is a named FOLLOW-UP stage item (two implementations of one semantic is drift surface); the undeclared-effect (`produces`-set) refusal deliberately STAYS at the bridge commit seam — it is engine-manifest conformance at the commit boundary, not a capability-contract check (avoids duplicating the check this stage). |
| DR-8 | **EffectClass gate semantics:** `Pure` → zero proposed effects ENFORCED (refusal). `IdempotentEffect`/`ProposesWrite` → effects pass through as PROPOSALS only; routing them to the Kernel commit gateway is the caller's job (ORCH-001). External-effect idempotency remains a provider DECLARATION the fabric cannot verify (design §3.1.4 honesty caveat; RUNTIME_FREEZE v1.1 debt #2) — honestly NOT checked. |
| DR-9 | **Fixed authorization check order** (deterministic, documented on `authorize`): (1) active binding → (2) engine identity → (3) capabilities-required existence → (4) policy. The capabilities-required check is EXISTENCE of an active binding in the SAME shard only; the identity check applies to the invoked capability (a required capability's own provider identity is checked when IT is invoked). |
| DR-10 | **Revocation propagation is resolve-time only (no async invalidation):** an `Authorization`/`GatedInvocation` issued BEFORE a revoke keeps its pinned version (exactly the §3.8 rebind-during-flight rule applied to revocation); the revoke bites every SUBSEQUENT authorize/resolve. Distributed revocation propagation, cancellation of in-flight work (OQ-6/CAP-008 (PROPOSED)) and lease/TTL semantics are NOT built and NOT claimed. |

## HONEST SCOPE — what this is and is not

- **Is:** the single-process capability-fabric core of I4 — binding lifecycle with
  append-only supersession + revocation, authoritative/pinned resolution, and the
  capability-gated invocation path (authorization → RCR-012-enforced engine invocation →
  EffectClass validation), all in-memory, trusted-single-host (the v1.0 threat model).
- **Is NOT (nothing beyond this is claimed):** no scheduler, no placement, no admission
  control/backpressure (IDR-007-gated, design §6), no selection policy (Control Plane),
  no decision-trace emission (OQ-10), no policy ENGINE (verdicts are Governance-supplied
  inputs), no approval/HITL sequencing, no distributed registry/replication of bindings,
  no distributed revocation or cancellation (OQ-6), no disk persistence of bindings
  ("durable semantics" = the append-only supersession HISTORY, not disk durability —
  bindings are configuration, the fabric persists nothing, per the frozen crate doc),
  no cryptographic authN/authZ (capability/policy gating only — RUNTIME_FREEZE #8 v2.0
  debt; design §3.16), no provider-artifact integrity verification at bind/dispatch
  (OQ-8). The RCR-012 determinism probe remains a PROBE, not a proof.
- **CAP-001..009 remain PROPOSED** — cited throughout only as (PROPOSED — CCP-GATE
  required); nothing here enforces them as conformance criteria. Registered bases used
  instead: Vol 9 Part 3 (selection vs carrying), ORCH-001/003/004, OWN-001, SHARD-001.

## Invariants (registered only)

- **OWN-001** — the fabric (registry instances) stays the single owner of binding state
  and of nothing else; the gate is stateless; nothing outside the registry mutates a
  chain (append-only, refused ops provably perturb nothing).
- **LAYER-001** — ONE new downward edge capability(70) → engine(60); ranks checked in
  `property_check.rs` before adding; executable architecture gate green over the
  workspace; no upward or lateral edge.
- **SHARD-001** — lifecycle chains and gate authorizations are scoped by the immutable
  shard key; cross-shard isolation proven at both the registry and the gate.
- **ORCH-001** — the crate has NO kernel dependency: neither the registry nor the gate
  can commit truth (structural); effects leave the gate as PROPOSALS only.
- **ORCH-002** — no plans, no persistent decision state; the gate's `Authorization` is a
  per-call proof token, held by the caller, reconstructible from registry state.
- **ORCH-003** — pinned-version replay surface: the binding that actually ran stays
  reconstructible (`resolve_pinned` + `history`) across supersession AND revocation.
- **ORCH-004** — the gated path verifies the fabric-derived content-addressable
  idempotency key on every invocation (RCR-012 `invoke_enforced` wired in).

## Adversarial review revisions (s1-fabric-core reviser pass, 3 minor findings, all applied)

1. **Mis-citation fixed (`lifecycle.rs::revoke` doc):** version monotonicity of the
   supersession chain was wrongly attributed to ORCH-004 (which governs idempotent,
   content-addressable INVOCATIONS). Re-anchored to its real basis: the frozen
   `RegistryError::NonMonotonicVersion` contract + design §3.5 step 1. The frozen
   `lib.rs` contract text is byte-unchanged.
2. **Design resolution — any `resolve` refusal is F-UNBOUND by definition
   (`gate.rs::authorize`):** the gate now matches on the `RegistryError` variant,
   preserving the registry-reported `capability`/`shard` fields for `Unbound`, and
   documents on `GateDenial::Unbound` that a foreign `CapabilityRegistry`
   implementation returning any other variant from `resolve` denies under this same
   class (shared meaning: "no active binding was resolvable"). In-repo registries
   return only `Unbound` from `resolve`; no denial path weakened.
3. **Manifest-stability seam documented (`invoke_gated` doc):** `authorize` checks one
   `engine.manifest()` snapshot while `invoke_enforced` (RCR-012) re-fetches the
   manifest for ORCH-004 key derivation; the gate ASSUMES manifest stability across
   those calls — trusted-single-host v1.0 threat-model caveat (design §3.16), same
   honesty class as the determinism-probe note. Threading one snapshot through both
   steps is a later-stage improvement, not done here.

Post-revision verification: `cargo test --workspace` → **193 passed / 0 failed**
(doc/traceability + error-mapping changes only; test count unchanged from Stage 1).
