## Design resolutions (open points; simplest option consistent with IDR-001..005 — recorded, never silently invented)

| # | Resolution |
|---|---|
| DR-1 | **Home crate = `arves-kernel`** (see Cargo.toml row above): LAYER-001 permits only kernel(40)→consensus(30); the Raft core stays in `arves-consensus` (design §3.20), the gateway wiring lives with the gateway (§6.2 row 2). |
| DR-2 | **Replica-mode Kernel = configuration of `RefKernel`, not a new trait (OQ-10):** each replica owns a private `RefKernel<MemWalStore>`; "replica mode" is the fact that `ClusterSim` holds it privately and the ONLY write door is the leader's `ClusterKernel::commit` — followers are reached exclusively by the apply loop. Confirms the I2.0 §9 assumption the design flagged for RCR review. |
| DR-3 | **Replicated payload = the `ProposedWrite`, length-prefixed LE-encoded** (same codec discipline as the Kernel snapshot blob); decode is strict (truncation/trailing garbage refused loudly at apply). Wire format for REAL transport stays OQ-3 → IDR; this is the in-process representation only. |
| DR-4 | **The two frozen `ContentHash` shapes stay separate (OQ-2 open):** the consensus digest carries a deterministic hex of the kernel content bytes and is NEVER load-bearing — identity checks (admission, DR-8) compare the full decoded proposal. Binding to ACS-001 remains the OQ-2 IDR. |
| DR-5 | **RCR-019 DR-8 discharged:** after quorum, `commit` verifies the entry at the awaited index is ITS encoded proposal; a foreign entry (legal after failover) surfaces as `NotReplicated` — retriable, and ORCH-004 keeps the retry idempotent. Likewise a pending entry left by a `NotReplicated` return either commits later (a retry then resolves `AlreadyCommitted` to the same truth) or is truncated by the successor (no partial truth, A-005/A-006) — at-least-once client semantics converge to exactly-once truth. |
| DR-6 | **Gateway semantics preserved by SHARING, not forking:** the validation head of `commit_inner` was factored into `admission` (used by single commit, RCR-013 batch validation path unchanged, and the cluster pre-propose check); follower apply calls the unmodified `RefKernel::commit`. A committed fork attempt (same address, different payload, distinct log entries) is refused IDENTICALLY on every replica — a deterministic skip, not a panic — so truth never diverges (RCR-005 under replication). |
| DR-7 | **Membership entries are not truth:** the apply loop skips `EntryKind::Membership` (consensus mechanism state, design §3.6 kind 2); WAL offsets stay dense and identical across replicas because ONLY outcome applies append. |
| DR-8 | **Snapshot install materializes the blob into the target's durable WAL as a dense continuation** (skip below head, assert-no-gap at head): required so post-snapshot applies assign the SAME `CommitIndex`es as peers (IDR-002 "identical truth" includes identical indexes). The WAL `term` field stays 0 exactly as `commit_inner` writes it everywhere (term wiring belongs to the IDR-005 unification stage). |
| DR-9 | **Crash model scope:** `crash_recover` loses/recovers KERNEL truth via local-WAL replay (I1.7); raft in-memory state is NOT crash-modeled — its durability is the IDR-005 WAL-unification stage. Apply cursors survive because the WAL is in lockstep with applied truth; even a stale cursor would only cause idempotent re-applies (ORCH-004). |
| DR-10 | **No cluster `commit_batch`:** the design does not cover it (§3.15 explicitly separates log batching from atomic multi-effect commit; §6.1 lists Kernel batch-commit as a non-goal with its own instrument). RCR-013's batch remains single-node; "a Raft entry naturally carries a batch" stays future work per RCR-013's own note. |
| DR-11 | **Error mapping within the frozen `CommitError`:** consensus `NotLeader`/`ElectionInProgress` → `NotLeader{shard}` (an absent-leader window means THIS node may not commit — the frozen enum's honest shape); `QuorumUnavailable` → `NotReplicated`; commit-wait budget is the deterministic `COMMIT_WAIT_MAX_TICKS = 200` logical ticks (a fixed budget, not a wall clock; real timing is the OQ-4 IDR). |
| DR-12 | **`truth_hash` is the single-shard cross-node instrument; per-shard state blobs are the general one:** across MULTIPLE shards the fold order of a node's committed vector legally differs with apply interleaving, so cross-node equality is asserted on `snapshot_shard` BYTES per shard (offset-ordered, deterministic — a byte-level truth-state comparison in the direction OQ-11 asks for). |
| DR-13 | **In-memory vehicle:** replicas use `MemWalStore` (deterministic, Arc-shared "durability" across crash simulation) — the same substrate/logic split I1 proved (`RefKernel` is generic over `WalStore`; `FileKernel` runs identical logic). Multi-host file-backed operation is transport-era work. |
| DR-14 | **Kernel wiring binds to the in-process sim vehicle, NOT (yet) the frozen `ShardConsensus` trait:** `ClusterKernel`/`ClusterSim` call `SimCluster`'s concrete API (`propose`/`tick`/`commit_of`/`log_of`) directly, bypassing the design's central `ShardConsensus` contract (§1.3/§2) and the RCR-019 `await_commit` path. Reason: the deterministic apply loop and the DR-8 identity check need log/commit-index introspection and scripted-tick control the frozen trait deliberately does not expose (it is a production contract, not a harness contract). Consequence honestly stated: the contract's first impl (`SimShardConsensus`) is NOT exercised by this Kernel path (it keeps its own RCR-019 conformance tests). Rewiring the Kernel commit path through `ShardConsensus` (propose/await_commit) is a RECORDED OBLIGATION of the transport/persistence-wiring stage, where the trait is the only surface a remote peer can offer. |

## HONEST SCOPE — what is simulated vs real

- **In-process-simulated:** transport (FIFO bus), faults (bus filters), time (injected
  logical tick), durability medium (in-memory Arc-shared WAL). NO network exists; NO
  network fault-tolerance is claimed (design §3.7: sockets arrive at the step where they
  are the property under test).
- **Real:** the leader-only commit gateway over real quorum replication; the follower
  apply loop producing byte-identical per-shard truth; ORCH-004/RCR-005 gateway semantics
  preserved under replication; Kernel snapshot install + log-tail catch-up — the install
  MECHANICS (state blob + dense WAL continuation + cursor jump, byte-identical result)
  are real and proven, but the TRANSFER IS ORCHESTRATED BY THE TEST HARNESS
  (`ClusterSim::install_snapshot` is invoked explicitly by tests), NOT initiated by the
  Raft leader; protocol-driven snapshot install (with raft-log compaction, OQ-1) is a
  later stage; CP refusals
  (`NotLeader`, `NotReplicated`); crash-recovery by deterministic local replay — all under
  the RCR-019 harness that re-checks the four Raft safety properties after EVERY message
  step of every test.
- **Not in this stage:** IDR-005 unification (the raft log is still the in-memory core
  log — RCR-019 DR-4; each replica's durable WAL receives applied outcomes; making the
  raft log itself the durable WAL is the persistence-wiring stage), raft-state crash
  durability, raft-log compaction (OQ-1), read tiers (I2.9), wire format/transport (OQ-3),
  cluster batch commit (DR-10), shard placement (OQ-8/I4).

## Invariants (registered only)

- **OWN-001** — one owner per state under replication: the leader is the sole writer per shard (`cluster_follower_commit_refused_not_leader`); replica kernels are private — no second door; consensus metadata stays consensus-owned.
- **LAYER-001** — the ONE new edge kernel(40)→consensus(30) is downward-only; the executable architecture gate (`property_check_suite_holds`, `architecture_gate`) re-verified green over the real Cargo graph.
- **SHARD-001** — routing strictly by immutable `(tenant, workspace)`; `UnknownShard` for unrouted keys; apply asserts no cross-shard bytes in a shard's log; no cross-shard commit API exists.
- **ORCH-001** — consensus carries the outcome opaquely (encode/decode live only in the Kernel's cluster module); only the Kernel-through-leader path creates truth; no state delta without a committed log entry (quorum-loss test: zero partial truth).
- **ORCH-003** — follower apply and recovery replay from the recorded trace, never recompute; cross-node determinism proven by byte-identical shard-state blobs + the identical-seed cluster run test; crash recovery = local-WAL replay.
- **ORCH-004** — idempotent, content-addressed commit survives distribution: duplicate re-proposal resolves to the SAME `TruthRef` on the leader and dedupes identically at every replica's apply; retry after `NotReplicated`/`NotLeader` converges to exactly-once truth.
- ORCH-002 — untouched (no Control-Plane surface; the replicated log taxonomy remains Outcome|Membership by the frozen `EntryKind`).
