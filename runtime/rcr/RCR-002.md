# RCR-002 — Truth-store tamper-evident integrity digest (Runtime v1.1)

**Status:** Applied (v1.1) · **Type:** additive, non-breaking · **Gate:** `cargo test --workspace`
must stay green (was 69) → **71 passed / 0 failed** after this change.

## Motivation

The Build Program closure audit (freeze backlog item #8) found that the WAL/snapshot store uses
**CRC32** (error-detection, *forgeable*) with no hash chain, Merkle root, or signature: an actor
who edits a committed record can recompute the per-frame CRC and the tamper passes recovery
undetected. This is a maintainer-directed, **proactive** v1.1 hardening (not incident-driven) —
authorized as a planned RCR, executed through the sanctioned Runtime Change Request path, not a
silent edit.

## Change (what shipped)

`runtime/crates/arves-persistence`:
- A **dependency-free SHA-256** (FIPS 180-4) — no external crates (the workspace has zero
  non-`arves-*` dependencies), deterministic across runs/platforms.
- **`FileWal::integrity_digest() -> [u8; 32]`** — folds a SHA-256 hash-chain over the committed
  record trace: `d₀ = SHA256(genesis(shard))`, `dᵢ = SHA256(dᵢ₋₁ ‖ bodyᵢ)`. Any alteration of any
  committed record changes the digest, **including a tamper that repairs the per-frame CRC32**.

The change is **read-only and additive**: it does **not** modify the WAL frame format, `append`,
`decode_all`, recovery, or checkpoints — so all pre-existing behavior (and the 69 prior tests) is
unchanged. No public API was removed; one method was added.

## Verification

- `cargo test --manifest-path runtime/Cargo.toml --workspace` → **71 passed / 0 failed** (the 69
  prior tests unchanged; +2 new).
- New regressions (`rcr002_integrity`):
  - `integrity_digest_is_deterministic_across_reopen` — a fresh store over the same durable root
    recomputes the identical digest.
  - `integrity_digest_detects_crc_fixed_tamper` — flips a payload byte of a committed record **and
    repairs its CRC32** so recovery still accepts it (`head == 5`), yet the digest **changes** —
    proving detection of the exact attack CRC alone cannot see.

## Scope (honest — what this is and is NOT)

- **IS: tamper-EVIDENCE.** With the digest anchored by a trusted holder (the Kernel, or a
  checkpoint records the expected value and compares later), any committed-record tamper is
  detectable; a single-record edit now forces rewriting the entire downstream chain.
- **IS NOT: tamper-PROOFing against a hostile host.** A fully compromised host that rewrites the
  whole trace *and* the anchor cannot be stopped by a hash-chain alone. That requires cryptographic
  **signatures** + an **authenticated commit path** (principal/authN on `Kernel::commit`) + durable
  **anchoring** — a **v2.0** change, outside v1.0's *trusted-single-host* threat model. This RCR
  adds the chain those signatures will sign; the signing/authN/anchoring remain open (freeze #8).

## Follow-ups (still open)

- v2.0: sign the digest; authenticate `Kernel::commit` (principal/authZ/tenant-scope); anchor the
  head digest durably (checkpoint/Kernel) and verify on recovery.
- v1.1 (optional): have the Kernel record + compare `integrity_digest` at checkpoint boundaries so
  tamper is caught automatically on recovery, not only on explicit audit.
