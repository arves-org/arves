> §5.4.2 term guard + RCR-019 DR-2's no-election-no-op keep prior-term entries out of the

> DR-2), so a `/`-bearing tenant's gather could serve another tenant's shard under the

## Design resolutions (open points; simplest option consistent with IDR-001..005 — recorded, never silently invented)

| # | Resolution |
|---|---|
| DR-1 | **Read-index, in-process form (design §3.7 "Through leader (read-index)"):** the router consults the sim-omniscient shard directory (`ClusterSim::leader_of` = highest-term live leader), takes that leader's raft commit index as the read-index, and serves only at a replica whose applied index ≥ it — exact under the single immutable borrow of the sim (nothing advances mid-read). HONEST LIMIT: the raft leadership-confirmation heartbeat round is NOT modeled (no network exists); the omniscient directory closes the stale-leader hazard that round exists to close. A networked read-index is a later stage. |
| DR-2 | **Bounded-staleness attestation degenerate (OQ-2 IDR still pending):** the only lag this stage can PROVE in milliseconds is zero (replica applied ≥ leader commit ⇒ 0 ms ≤ any bound, clock-free). Any other state is refused with `observed_lag = LAG_UNATTESTABLE (= Millis::MAX)` — an explicit "unknown, treat as unbounded" sentinel, deliberately NOT a measured value: inventing a time↔index mapping here would be the exact silent-violation risk the design's Risk-2 names. The general mechanism awaits its own IDR before any claim of real bounded reads. |
| DR-3 | **OQ-4 (partial scatter-gather) → fail-WHOLE:** any sub-read failure fails the entire gather with that `QueryError`, verbatim; no partiality label exists in the frozen enum and none is fabricated. Per-shard `NotFound` is NOT a failure (that shard contributes to the version vector only — union semantics). OQ-3 is resolved to "yes, tenant-internal fan-out ships" by the milestone mandate; cross-tenant scatter stays structurally impossible (the API takes one tenant; no tenant-set overload exists). |
| DR-4 | **Additive-surface shape (design §3.3/§6.2, OQ-5):** the frozen single-shard `Query` trait is untouched; the fan-out entry point (`gather_read`), merged type (`GatheredRead` with per-shard version vector, NO global version), floor carrier (`read_at_least`, `floor_of`) and floor error (`FloorReadError` — a new TYPE, never a new variant of the frozen `QueryError` enum, which would break exhaustive matches = v2.0) land as additive v1.x surface via this RCR, exactly as §6.2 rows 5–7 prescribe. A tenant with zero registered shards is refused as `UnknownShard` carrying the bare tenant text (least-wrong frozen variant, RCR-023 DR-7 house style). |
| DR-5 | **Registered-but-empty shard at a replica:** a shard present in the directory whose WAL does not yet exist at the serving replica is served as the EMPTY fold at position 0 (zero committed truth here — honest), not an error; `MemWalStore::open`'s create-on-demand is never triggered by a read (existence is checked against `shards()` first — reads stay pure observations). |
| DR-6 | **Routing inputs vs Kernel read hook (ORCH-001 preserved):** `arves-query` now depends on `arves-kernel`/`arves-consensus` for exactly three read-only routing inputs (directory listing, leader identity + commit/applied raft indexes, replica WAL-store handle) — the design's §3.4 rows 2/4 downward edges. NO Kernel truth accessor is consumed (`truth_hash_of`/`shard_state_of`/`snapshot_shard` untouched by library code); served bytes come from WAL replay only. The RCR-023 "no kernel edge" stance was the Stage-1 degenerate; the design itself names these edges for the distributed stage. |
| DR-7 | **Fan-out execution:** deterministic SEQUENTIAL sub-reads in ascending shard order under one borrow of the sim. Design §3.8's concurrent scatter with join deadline is a distributed-transport concern; no concurrency is claimed in-process (honest-language rule). |
| DR-8 | **Read-index PRECONDITION (revision; interaction with RCR-019 DR-2):** DR-1's honest-limit covered only the leadership-confirmation heartbeat round — NOT the second Raft §6.4 precondition: a read-index is valid only once the leader has a COMMITTED entry of its CURRENT term. Because RCR-019 DR-2 deliberately appends no no-op on election and `advance_commit` carries the §5.4.2 term guard, a freshly elected leader's commit index can EXCLUDE prior-term quorum-committed (acked) entries; taking it as the read-index admitted Linearizable/zero-lag-Bounded reads that silently missed acked writes (proven by an executed PoC schedule, now the regression test). Resolution: `admit` refuses both strong tiers (`LeaderUnavailable`; `StalenessBoundExceeded`+`LAG_UNATTESTABLE`) unless `has_committed_in_current_term(leader, shard)` holds — empty log trivially current. CP-honest availability-only degradation; the AP tier is untouched; no frozen surface changed. |
| DR-9 | **Typed gather routing (revision; RCR-023 DR-2 lineage):** the fan-out already held the exact `ShardId` but round-tripped it through the ambiguous `"tenant/workspace"` text (first-`/` parse; `/` is legal inside a part), so a gather for a `/`-bearing tenant could serve ANOTHER tenant's shard under the wrong label — the RCR-023 DR-2 ambiguity re-created on a surface that constructs the text ITSELF. Resolution: `serve` is split into a typed core (`serve_typed`) taking `(ShardId, WalShardKey)`; text parsing exists ONLY at the frozen `ReadScope` entry points (which keep DR-2's documented behaviour); gather sub-reads route on the typed identity, and the "structurally impossible" claim is softened to the provable statement (typed-tenant equality; text keys are display labels, never routing inputs). Mitigants unchanged: v1.0 trusted single host, no fold ever mixes two shards' records. |

## HONEST SCOPE — what this is and is not

- **Is:** the distributed READ path of I3 over the deterministic in-process I2 cluster:
  real shard-aware routing, real per-replica WAL-replay projections, real CP/AP tier
  behaviour (linearizable refusal under partition/leader loss; labeled AP staleness),
  real tenant-internal fan-out with union semantics — all under scripted faults and
  logical ticks. Truth stays CP (only the leader-gated Kernel commits); this surface is
  the AP observability side and *labels* what it serves.
- **Is NOT:** a networked query fabric. No wire, no sockets, no concurrent readers
  racing appliers, no real read-index heartbeat round (DR-1), no real millisecond lag
  attestation (DR-2 — `StalenessBoundExceeded` here always carries the sentinel), no
  protocol-driven snapshot bootstrap of query replicas, no LCW read views (OQ-8), no
  authN/authZ (OQ-1 — trusted single host, as at v1.0), no query language/planner
  (design §6.1 #5), no cross-shard atomic or snapshot-consistent reads (§6.1 #3 — the
  version vector makes skew visible instead of hiding it).
- **Consistency, stated exactly (DR-8 revision wording):** *leader-consistent read* =
  linearizable w.r.t. quorum-committed truth, served ONLY when the attesting leader's
  read-index is VALID (a committed entry of its current term exists — Raft §6.4); a valid
  read-index provably includes every acked commit, and the served fold may reflect newer
  commits, never older ones. Without a valid read-index the read REFUSES — it is the
  refusal, not the directory, that closes the new-leader commit-index-lag hazard.
  *Follower/AP read* = the serving replica's applied prefix, arbitrarily
  stale under partition, always labeled with its true `observed_at` and `served_tier`,
  never wrong for the version it reports. *Bounded read* = zero-lag-proven against a
  valid read-index, or refused.
- **QUERY-001 remains PROPOSED** — cited only as proposed; read-only-ness is enforced
  via the registered A-003 Layer-Matrix row, the frozen trait shape, and structure
  (immutable sim borrows; no path to `Kernel::commit`; WAL consumed behind `&`).

## Invariants (registered only)

- **OWN-001** — Query owns only derived, disposable folds; truth's one owner stays the
  leader-gated Kernel; reads change zero truth on every replica (proven).
- **LAYER-001** — two new downward edges (60→40, 60→30); architecture gate green; no
  upward or lateral edge anywhere.
- **SHARD-001** — every read resolves exactly one shard; scatter is single-tenant with
  typed-`ShardId` sub-read routing (DR-9 — never re-parsed text); cluster-wide two-tenant
  negative proof on every replica × tier, plus the `/`-bearing-tenant regression.
- **ORCH-001** — reads reconstruct committed truth from replica WALs only; the new
  ClusterSim accessors expose routing metadata and the Persistence substrate, never
  Kernel truth; nothing uncommitted is visible (only committed entries are applied to
  WALs; the un-acked minority suffix of RCR-021's model is never applied anywhere).
- **ORCH-003** — distributed replica-equality: converged replicas serve identical
  projections (same fold, same digest basis as RCR-023 proved single-node).
- **ORCH-004** — reads idempotent (identical results, zero state change cluster-wide);
  `ProjectionId` stays the content-addressed hex id.
- ORCH-002 — untouched (no Control-Plane surface in this stage).
