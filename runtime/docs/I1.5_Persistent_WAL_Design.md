**Milestone:** I1.5 (sub-step of frozen Baseline `I1 Distributed Runtime`)
**Status:** Design → Implementation
**Predecessor:** I1.4 First Executable Behaviour (`tag: I1.4-first-behaviour`)
**Governing:** ARVES Engineering Constitution (CLAUDE.md); frozen specification is NOT modified.

> This document executes the mandatory workflow phases 1–10 for I1.5 before any

## 0. One-line intent

Make the Write-Ahead Log **real**: a committed record is `fsync`'d to disk
**before** `append` returns, survives a full process exit, and is recovered
**byte-identically** by a fresh process that only knows the directory path.
Proof target: **Memory → Disk → Restart → Recovery.**

---

## 1. Architecture Readiness Review

I1.4 proved the commit→WAL→replay behaviour over an **in-memory** substrate
(`MemWalStore`), where "restart" was simulated by dropping the Kernel while an
`Arc`-shared `Vec` kept the log alive. That is durability *theatre*: nothing
touched the disk, so a real crash would lose everything. I1.5 replaces the
substrate with a genuine on-disk log. Nothing above Persistence changes; no new
architectural layer is introduced (Non-Negotiable Rule #3). Ready.

## 2. Affected UCI node

**Persistence layer** (Data Plane), and the wiring in the Kernel that opens/
replays a store. No change to LCW, Query, Engine, Capability, Execution, or the
Control Plane.

## 3. Specification mapping

- **IDR-005** — Raft log = WAL = decision trace, **append-only**. The file WAL is
  the durable face of that single artifact; records are only ever appended.
- **IDR-002** — replicate/persist committed **outcomes**, not invocations. WAL
  frames carry committed outcomes (`RecordKind::Outcome`).
- **ORCH-003 / IDR-003** — recovery is **replay from the recorded trace**, never
  recomputation. `replay_from` re-reads the on-disk frames in offset order.
- **SHARD-001** — partition by tenant/workspace; immutable key. One file per
  shard; offsets are per-shard and never comparable across shards.
- **ORCH-001 / OWN-001 / PERSIST-001 (proposed)** — the store owns **no truth**;
  it persists what the Kernel commits and never mutates a committed record.
- **LAYER-001** — Persistence depends on nothing above it (std only; no deps).

## 4. Contract mapping

Implements the **existing, unchanged** traits `Wal` / `WalStore` / `ReplayCursor`
from `arves-persistence`. The `Wal::append` doc already states: *"The record must
be durable (survive crash) before this returns Ok."* I1.5 finally honours that
sentence with a real `fsync`. No trait signature is added, removed, or altered.

## 5. Invariant mapping

| Invariant | How I1.5 upholds it |
|---|---|
| IDR-005 append-only | File is opened `append`-only; existing frames are never rewritten; compaction (prefix drop) is deferred to I1.6. |
| ORCH-003 replay | `recover()` reads frames from offset 0 in file order; truth is reconstructed, never recomputed. |
| ORCH-004 idempotent | Kernel still de-dupes by `(shard, content)`; a duplicate commit writes no second frame. |
| SHARD-001 | One file per shard, `hex(tenant)__hex(workspace).wal`; `append` rejects a record whose shard ≠ the file's shard. |
| OWN-001 / ORCH-001 / PERSIST-001 | Store performs no validation of meaning; it frames bytes, checksums them, fsyncs, and reads them back. |

## 6. Ownership analysis

The Kernel remains the **sole** truth owner and commit gateway. The store is the
Kernel's durable substrate, not a second owner. To let the Kernel run over either
substrate without duplicating logic, the concrete Kernel becomes **generic over
`S: WalStore`** (`RefKernel<S>`); `MemKernel` and `FileKernel` are type aliases.
This changes *no ownership* — it removes a hard-coded dependency on one store
impl, which is the opposite of coupling.

## 7. IDR mapping

IDR-001..005 all concern the **distributed** log (Raft, quorum, membership,
leader election). **None** of that is in scope for I1.5 — this is single-node
durability. The file WAL is deliberately shaped so the *same artifact* can later
become the Raft log (offset = log index, term field already present) without a
format change. No IDR is violated; none is implemented early.

## 8. Gap analysis

| Have (I1.4) | Need (I1.5) |
|---|---|
| In-memory `Vec` log | On-disk append-only file per shard |
| No durability | `fsync` (`sync_all` = FlushFileBuffers on Windows) before `append` returns Ok |
| No corruption handling | Length-framed + CRC32 frames; torn tail detected & truncated |
| Kernel hard-wired to `MemWalStore` | Kernel generic over `WalStore`; `FileKernel` alias |
| "Restart" = drop Kernel, Arc keeps log | Restart = **new process / new store** reads the directory |

## 9. Engineering design

**On-disk format (per shard, one file).** A sequence of self-describing frames:

```
frame := u32 body_len (LE) | body[body_len] | u32 crc32(body) (LE)
body  := u8 version(=1) | u8 kind | u64 term | u64 offset
       | u32 len|tenant | u32 len|workspace | u32 len|content | u32 len|payload
```

- **Deterministic & dependency-free:** fixed field order, little-endian, a
  hand-rolled CRC32 (IEEE, reflected) and hex filename encoding — no external
  crates (auditability; determinism; LAYER-001).
- **Crash-consistency:** on `open`, frames are decoded in order; the first torn
  (short) or CRC-mismatched frame ends recovery. The good-prefix byte length is
  used to `set_len` the file, truncating the garbage tail so future appends
  continue from the last durable record. Truth committed before the tear is
  intact.
- **`append`:** build frame at `offset = head`, one `write_all`, then `sync_all`
  (durability obligation), then `head += 1`, return `offset`.
- **`replay_from(o)`:** read the whole file, decode good frames, hand back
  `[o..]` via a `VecReplayCursor` (shared with `MemWal`).
- **`FileWalStore`:** `root/<hex tenant>__<hex workspace>.wal`. `open` caches an
  `Arc<Mutex<inner>>` per shard within a process; a **fresh store instance** over
  the same root has an empty cache and rebuilds `head` by scanning the file —
  that scan *is* the recovery path. `shards()` enumerates the directory (so a new
  process discovers shards written by a dead one).

**Restart model (the real upgrade over I1.4):** process A commits + `fsync`s +
exits; process B, a genuinely separate OS process (`std::process::Command`),
opens the directory, replays, and computes the same `truth_hash`. Within-process
tests additionally drop all handles and re-open a *new* store to prove disk (not
`Arc`) round-trips.

**Concurrency:** one `Mutex` per open file guards `head` + the append handle;
`replay_from` uses an independent read handle over the fsync'd bytes.

**Cross-shard ordering — explicit non-goal.** Per SHARD-001 offsets are
per-shard and never comparable across shards, so there is **no** global total
order across shards and `truth_hash` is only order-stable within one shard.
Behaviour proofs that compare a global hash use a single shard; the multi-shard
proof checks each shard recovers its own frames independently. A cross-shard
deterministic order is a consensus/query concern for later milestones, not a
promise Persistence makes.

**Failure modes & recovery:** I/O error → `WalError::Durability(_)`; torn tail →
truncated on open; wrong-shard append → `WalError::UnknownShard`; read past head
→ `WalError::OffsetOutOfRange`.

**Known limitation (honest):** we `fsync` the file but not the parent directory
entry on first create (std has no portable dir-fsync; on Windows `sync_all` =
FlushFileBuffers on the file). A power-loss between file-create and first append
could in theory lose a just-created empty file on some POSIX filesystems. This is
acceptable for the single-node reference proof and is noted for the durability
hardening milestone. No truth is lost because no truth exists until the first
`append` returns (which does fsync the file's data + size metadata).

## 10. Critical self-review (attempt to break it)

- *Torn write mid-frame?* Length prefix + CRC over body ⇒ detected; tail
  truncated; prior truth intact. ✔
- *Bit-flip in a committed frame?* CRC mismatch ⇒ that frame and everything after
  it drop. This is conservative (we never serve corrupt truth) but means a
  mid-log flip truncates the suffix. Acceptable at I1.5 (single-node, no
  replication to repair from); replication-based repair is a later, distributed
  concern. Documented, not hidden. ✔
- *Non-deterministic replay?* Fixed-order LE encoding + file-order iteration +
  `shards()` sorted ⇒ deterministic. No `HashMap` iteration, no clock, no RNG. ✔
- *Offset drift after recovery?* Decoded frame's stored `offset` is validated
  against its position; mismatch ⇒ treated as corruption. ✔
- *Coupling from making the Kernel generic?* It removes a concrete dependency;
  aliases preserve the I1.4 surface verbatim. ✔
- *Spec drift?* No trait changed; no layer added; every behaviour maps to an IDR/
  invariant above. ✔

**Verdict: design PASS — proceed to implementation.**
