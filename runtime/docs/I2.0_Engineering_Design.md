**Milestone:** I2 Cluster Kernel (frozen Baseline Part 5).
**Property proven (ED-002):** **Replication.**
**Status:** Design → **awaiting ratification** (no code yet).
**Predecessor:** I1 Distributed Runtime COMPLETE (`tag: I1-complete`).
**Governing:** Engineering Constitution; RT-001 (interface evolution); ED-001/002/003;

## 0. One-line intent

Prove **replication semantics** and nothing more (yet): a **Follower** that
applies the **Leader's** committed outcome stream reconstructs **byte-identical
truth**. Definition of Done:

```
Leader append → replicate(committed outcomes) → Follower apply
             → follower.truth_hash() == leader.truth_hash()
```

Explicit non-goals for the first deliverable (deferred, see §11): leader
election, quorum, membership, failover, split-brain, and networking.

## 1. Architecture Readiness Review

I1 proved a single node can hold durable, replayable, recoverable truth. I2 asks
the next question: can a **second** node hold the **same** truth by *applying the
leader's committed outcomes* — without re-deriving them? That is
state-machine replication reduced to its essence, minus consensus. I1 already
shaped the substrate for it: the WAL *is* the replicated decision trace (IDR-005),
records *are* committed outcomes (IDR-002), and `replay_from` already streams the
committed trace in order. Ready.

## 2. Affected UCI node

**Consensus/replication fabric** (the `arves-consensus` crate, currently a
skeleton) plus the Kernel↔WAL boundary it consumes. No new architectural layer:
replication is a property of the cluster kernel over the existing
Kernel/Persistence, not a new tier. LCW/Query/Engine/Capability/Execution/Control
Plane unchanged.

## 3. Specification mapping

- **IDR-002** — replicate committed **OUTCOMES, not invocations**. The Follower
  applies `WalRecord`s (committed outcomes); it never re-runs the commit/derive
  path. ✔
- **ORCH-003 / IDR-003** — the Follower reconstructs truth by **applying the
  recorded trace**, never recomputing. Identical to recovery, but the trace
  arrives from the leader's stream instead of local disk. ✔
- **IDR-005** — Raft log = WAL = decision trace. The replication unit is exactly
  a WAL record; the leader's `replay_from` is the replication source. ✔
- **ORCH-001 / OWN-001** — the **Leader is the sole truth owner** for its shards
  and the only node that accepts client commits. The **Follower is a replica**:
  it applies the owner's decisions and never mutates truth independently, so
  there is still exactly one writer per shard. ✔
- **SHARD-001** — replication is **per shard** (one stream per tenant/workspace);
  offsets are per-shard log indices. ✔
- **IDR-001 (per-shard Raft, leader election)** — **NOT implemented yet**;
  I2.1 assumes a fixed, externally-designated leader (see §11 staging).

## 4. Contract mapping (RT-001 interface evolution — PROPOSED)

A minimal replication surface, justified under RT-001 (activating the reserved
IDR-002 "replicate outcomes" semantics; no frozen change). Sketch:

```rust
// Leader side: stream committed outcomes for a shard from an offset.
// (This is the leader's WAL replay surface, reused — the committed trace.)
trait ReplicationSource {
    fn changes_since(&self, shard: &ShardKey, from: Offset)
        -> Result<Vec<WalRecord>, WalError>;
}

// Follower side: apply a committed outcome, contiguously and idempotently.
// Appends to the follower's own durable WAL (preserving offset/term/content/
// payload) then inserts truth. A gap/out-of-order record is rejected LOUDLY
// (I1.7 discipline), never silently applied.
trait ReplicaApply {
    fn apply(&self, record: &WalRecord) -> Result<(), ReplicationError>;
}
```

The Follower reuses `RefKernel` + a durable `FileWalStore` at its own path; apply
= offset-preserving append + truth insert. No new frozen contract.

## 5. Invariant mapping

| Invariant | I2.1 obligation |
|---|---|
| IDR-002 | Replicate `WalRecord` outcomes; follower never re-derives. |
| ORCH-003 | Follower applies from the trace; identical to replay. |
| IDR-005 | Replication unit == WAL record; append-only on the follower too. |
| OWN-001 / ORCH-001 | Leader is sole writer/owner; follower is a read-only replica (rejects client commits). |
| SHARD-001 | One replication stream per shard; per-shard offsets. |
| ORCH-004 | Re-applying an already-applied record is an idempotent no-op. |

## 6. Ownership analysis

**Leader owns truth; Follower holds a replica.** OWN-001 (one owner per state) is
preserved because the follower never *decides* truth — it *applies* the leader's
committed decisions in order. There is no second writer: client commits go only to
the leader (IDR-002). The follower's copy is derived state, like a snapshot,
authoritative only as a faithful mirror of the leader's log.

## 7. IDR mapping / staging

I2.1 assumes a **fixed leader** (no election). This is deliberate: IDR-001
(per-shard Raft + election) and quorum/membership are consensus concerns that
layer on **after** replication semantics are proven. The WAL is already
Raft-log-shaped (`offset`=index, `term` present), so I2.5 (Raft log) is a
formalization, not a rewrite.

## 8. Gap analysis

| Have (I1) | Need (I2.1) |
|---|---|
| Single node commits + durable WAL | A leader whose committed records stream to a follower |
| `replay_from` reads local disk trace | Same trace consumed as a replication source |
| Recovery applies local trace | Follower applies a *remote-origin* trace (in-process transport for now) |
| Recovery rejects gaps loudly (I1.7) | Follower apply rejects out-of-order/gapped records loudly |
| `truth_hash` introspection | Cross-node equality: `follower.truth_hash() == leader.truth_hash()` |

## 9. Engineering design (I2.1 Replication Model)

- **Leader:** an ordinary `RefKernel<FileWalStore>`; commits as in I1. Its WAL is
  the committed decision trace.
- **Transport (I2.1):** **in-process** — the follower pulls `changes_since(shard,
  from)` from the leader (a function call over the leader's WAL). No sockets; this
  isolates *replication correctness* from networking (a separate later concern),
  exactly as I1.5 isolated durability from distribution.
- **Follower:** a `RefKernel<FileWalStore>` at its own path in **replica mode**
  (rejects client `commit`). It applies each streamed `WalRecord`:
  1. require `record.offset == follower_wal.head` for that shard (contiguous) —
     else `ReplicationError::Gap` (loud);
  2. append the record to the follower's WAL **preserving** offset/term/content/
     payload (offset-preserving append);
  3. insert the truth into the follower's set (idempotent by content).
- **Proof:** after the follower drains the leader's stream, for every shard
  `follower.truth_hash() == leader.truth_hash()` and `committed_count` matches.
- **Determinism:** apply strictly in offset order per shard; no clock, no RNG;
  identical bytes ⇒ identical `truth_hash` (same fold as I1).
- **Idempotency (ORCH-004):** re-delivering an already-applied record (offset <
  head) is a no-op, not an error; only a *gap* (offset > head) is loud.

**Failure modes:** gap/out-of-order → `ReplicationError::Gap` (loud); shard
mismatch → error; follower commit attempt → rejected (replica is read-only).

## 10. Critical self-review (to be hardened by the I2 adversarial hunt, ED-003)

- *Follower diverging from leader?* Apply is byte-for-byte from the leader's
  records; `truth_hash` equality is the executable guard. ✔
- *Silent gap on the follower?* Contiguity check rejects loudly (I1.7 culture). ✔
- *Two writers?* Follower rejects client commits; only the leader commits. ✔
- *Non-determinism?* Offset-ordered apply, no clock/RNG. ✔
- *Re-derivation drift?* Follower applies outcomes, never re-runs commit logic
  (IDR-002). ✔
- The full adversarial fault hunt (ED-003) runs before I2.1 is declared complete:
  lenses will include partial-stream apply, duplicate/re-ordered delivery, leader
  compaction vs follower catch-up (follower behind a leader checkpoint), and
  multi-shard interleaving.
