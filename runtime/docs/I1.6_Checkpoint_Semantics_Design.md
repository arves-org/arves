# I1.6 — Checkpoint Semantics — Engineering Design

**Milestone:** I1.6 (sub-step of frozen Baseline `I1 Distributed Runtime`)
**Status:** Design → **awaiting ratification** (no code yet)
**Predecessor:** I1.5 Durable WAL (`tag: I1.5-durable-wal`)
**Governing:** ARVES Engineering Constitution; frozen specification is NOT modified.

> Phases 1–10 below. Phase 11+ (implement/test/review) begins only after the
> **Decision Register (§D)** is ratified — because I1.6 evolves the *implementation*
> persistence contract, which our prior review scored as "no contract change".
> That posture is deliberate; changing it is a decision, not a default.

---

## 0. One-line intent

Stop the WAL from growing forever. Capture materialized state as a durable
**checkpoint**, drop the WAL prefix it covers, and make recovery =
**snapshot + replay tail** instead of replay-from-zero — without letting
Persistence ever interpret truth (ORCH-001) and without rewriting committed
frames (append-only, IDR-005).

## 1. Architecture Readiness Review

I1.5 proved durability but recovery replays **every** frame from offset 0; the
WAL is unbounded and restart cost grows linearly forever. That is the exact
problem checkpointing exists to solve. The type surface already anticipated it:
`SnapshotMeta`, `RecordKind::SnapshotMarker`, `Wal::snapshot`, `Wal::earliest`
are present but inert. I1.6 makes them real. Ready.

## 2. Affected UCI node

**Persistence layer** (durable checkpoint store + segmented WAL + compaction) and
the **Kernel↔Persistence boundary** (who produces/consumes the state blob). No
change to LCW, Query, Engine, Capability, Execution, or Control Plane. No new
layer.

## 3. Specification mapping

- **IDR-005** (append-only WAL) — committed frames are never rewritten.
  Compaction may only **drop a prefix already captured by a snapshot** (this is
  quoted verbatim in the current `Wal` trait doc). Segmenting the WAL lets us drop
  a prefix by **deleting whole covered segment files**, never by rewriting
  survivors. ✔
- **ORCH-003 / IDR-003** (replay, not recompute) — recovery loads the snapshot's
  materialized state, then replays only the WAL **tail**. Still reconstruction
  from the recorded trace + a derived checkpoint; never recomputation. ✔
- **ORCH-001 / OWN-001 / PERSIST-001** — the **Kernel owns truth**, so the Kernel
  must **produce** the state blob; Persistence stores it as **opaque bytes** and
  never interprets it. Snapshot metadata (offset/term/CRC) is durability data, not
  truth. This is the crux — see §6. ✔
- **SHARD-001** — checkpoints are **per shard**; a snapshot's `up_to_offset` is a
  per-shard offset, never comparable across shards. ✔

## 4. Contract mapping (implementation contract — PROPOSED evolution)

The current `Wal::snapshot(&mut self) -> SnapshotMeta` is too weak: it cannot
accept the Kernel's state blob and returns nothing to restore from. Proposed
**minimal, spec-justified** evolution of the `Wal` trait (implementation
contract, NOT the frozen UCS):

```rust
// Kernel hands over its materialized state for the shard (it owns truth,
// ORCH-001); Persistence stores the blob durably, records the recovery point,
// and MAY delete segments fully at/below up_to_offset.
fn install_snapshot(&mut self, up_to_offset: Offset, term: Term, state: &[u8])
    -> Result<SnapshotMeta, WalError>;

// Latest durable snapshot for restore, if any (opaque blob + its metadata).
fn load_snapshot(&self) -> Result<Option<(SnapshotMeta, Vec<u8>)>, WalError>;

// Drop WAL segments fully captured by a durable snapshot at/below up_to_offset.
fn compact(&mut self, up_to_offset: Offset) -> Result<(), WalError>;
```

- The existing `snapshot(&mut self)` marker method is **replaced** by
  `install_snapshot(..)` (it never had a real body — I1.4/I1.5 returned an inert
  marker). `earliest()` becomes **meaningful** (first retained offset after
  compaction); `replay_from(earliest())` yields the tail.
- Kernel gains two methods (it owns truth): `snapshot_shard(&self, shard) ->
  Vec<u8>` (serialize the shard's truths deterministically) and
  `install_state(&self, shard, &[u8])` (load a snapshot blob into the truth set,
  seeding the base offset). Public method `checkpoint(&self)` ties them together:
  produce blob → `install_snapshot` → `compact`.

## 5. Invariant mapping

| Invariant | Upheld by |
|---|---|
| IDR-005 append-only | Segmented log; compaction deletes fully-covered segments; survivors never rewritten. |
| ORCH-003 replay | Restore = load snapshot blob + replay tail; no recomputation. |
| ORCH-001 / OWN-001 / PERSIST-001 | Kernel produces & consumes the blob; Persistence treats it as opaque bytes, makes no decision about its meaning. |
| ORCH-004 idempotent | A re-run checkpoint at the same offset is a no-op (snapshot files are content/offset-named + CRC-checked). |
| SHARD-001 | One checkpoint + segment set per shard; per-shard offsets. |

## 6. Ownership analysis — the crux

**Who materializes the snapshot?** Persistence must NOT, because it owns no truth
(ORCH-001/PERSIST-001) and does not know the Kernel's state machine. Even though
our current Kernel's state is *trivially* "the set of distinct committed frames"
(so Persistence could technically fold the log itself), doing so would couple
Persistence to Kernel semantics and break the ownership split we have guarded
since I1.4. **Decision: the Kernel produces the state blob; Persistence stores it
opaquely and manages segment lifecycle.** This is the standard Raft
state-machine/log split and it keeps every invariant intact. (Rejected
alternative: log-only compaction where Persistence reduces the log — rejected on
ORCH-001 grounds even though it would "work" for today's trivial state.)

## 7. IDR mapping

IDR-001..005 concern the distributed log. I1.6 stays **single-node**: no Raft, no
quorum, no leader election. The segmented layout is shaped so a segment file can
later become a Raft log segment and `install_snapshot` can later back Raft's
InstallSnapshot RPC — **without a format change**. No IDR implemented early; none
violated.

## 8. Gap analysis

| Have (I1.5) | Need (I1.6) |
|---|---|
| One append-only file per shard | Per-shard **directory**: segment files + snapshot file(s) + manifest |
| `snapshot()` returns inert marker | `install_snapshot(offset, term, blob)` persists a real checkpoint |
| No compaction; WAL unbounded | `compact(offset)` deletes fully-covered segments |
| Recovery = replay from 0 | Restore = load latest snapshot blob + replay tail (`> up_to_offset`) |
| `earliest()` == 0 always | `earliest()` == first retained offset after compaction |
| Kernel replays only | Kernel can `snapshot_shard` / `install_state` / `checkpoint` |

## 9. Engineering design

**On-disk layout (per shard).** `root/<hex tenant>__<hex workspace>/` containing:
- `seg-<zero-padded-start-offset>.wal` — append-only segment files; a new segment
  is rolled when the current one crosses a size/offset threshold (**WAL
  rotation**). Frame format is the I1.5 format, unchanged (length + body + CRC32).
- `snap-<zero-padded-up_to_offset>.snap` — a checkpoint: header (up_to_offset,
  term, blob len) + opaque Kernel state blob + CRC32 over the whole file.
- Recovery-point selection is implicit: the highest `up_to_offset` valid `.snap`
  wins; older snaps and fully-covered segments are deletable.

**Snapshot (checkpoint) sequence** (Kernel-driven, crash-safe ordering):
1. Kernel serializes shard truths → `blob` (deterministic order + CRC).
2. `install_snapshot(N, term, blob)` writes `snap-N.snap.tmp`, `fsync`, atomically
   `rename` → `snap-N.snap`, `fsync` dir. **The snapshot is durable BEFORE any
   segment is deleted** (never lose truth on a crash between the two).
3. `compact(N)` deletes only segments whose **last** offset ≤ N. The segment
   containing offset N+1.. is retained. `earliest()` becomes the retained start.

**Restore (recovery) sequence:**
1. `load_snapshot()` → newest valid `.snap` (CRC-checked; a torn/partial snap is
   ignored, falling back to the prior snap or full replay — never serve corruption).
2. Kernel `install_state(shard, blob)` seeds truths + base offset = N+1.
3. Replay `replay_from(N+1)` over retained segments to apply the tail.
   Result must equal a from-zero replay (proven by test: checkpoint+compact then
   recover ⇒ identical `truth_hash`).

**Determinism & dependency-free:** snapshot blob uses the same fixed-order LE +
hand-rolled CRC32 as the WAL; segment/snapshot filenames are zero-padded decimal
offsets (sortable, deterministic); no new crates.

**Failure modes:** crash mid-snapshot → `.tmp` orphan ignored on restore (atomic
rename gate). Crash mid-compact → deleting a covered segment is idempotent; a
half-deleted set still has the snapshot, so restore is safe. Torn snapshot → CRC
fail → fall back. Compaction requested past head → `WalError::OffsetOutOfRange`.

**Trade-offs / honest limits:** atomic `rename` + best-effort dir `fsync`
(std has no portable dir-fsync; documented, as in I1.5). Segment size threshold
is a tuning knob, not a spec constant — kept small in tests to exercise rotation.

## 10. Critical self-review (attempt to break it)

- *Crash between snapshot write and segment delete?* Snapshot is fsync'd+renamed
  first; deletes are idempotent; worst case = extra retained segments replayed on
  restore (correct, just slower). ✔ No truth loss.
- *Restore drift (snapshot+tail ≠ full replay)?* Enforced by a behaviour proof
  comparing `truth_hash` of (checkpoint→compact→recover) vs (full replay). ✔
- *Persistence interpreting truth?* No — blob is opaque bytes produced/consumed by
  the Kernel only. ORCH-001 intact. ✔
- *Append-only violated by compaction?* No — we **delete** covered segments, never
  rewrite survivors. IDR-005 intact. ✔
- *Trait evolution = spec drift?* No — the frozen UCS (.docx corpus) is untouched;
  this evolves an *implementation* trait, justified by IDR-005/ORCH-003 and
  pre-anticipated by `SnapshotMeta`/`SnapshotMarker`. But because it flips our
  "no contract change" posture, it is surfaced for ratification (§D), not assumed.
- *Idempotent checkpoint?* Re-checkpoint at same N overwrites the identical
  `snap-N.snap`; re-compact is a no-op. ✔

**Verdict: design PASS pending §D ratification.**

## D. Decision Register — RATIFIED (maintainer, I1.6 design review)

**Decision: APPROVED (Option A).** Proceed with the segmented checkpoint
architecture. Treated as **Reference Runtime Interface Evolution under RT-001**,
NOT a modification of the frozen specification. See
`RT-001_Reference_Runtime_Interface_Evolution.md`.

| # | Decision | RATIFIED choice | Rejected alternative |
|---|---|---|---|
| D1 | Snapshot producer | **Kernel produces the state blob; Persistence stores it opaquely** (ORCH-001) | Persistence folds the log itself (couples layers; breaks ORCH-001) |
| D2 | Compaction mechanism | **Segmented WAL; delete fully-covered segments** (append-only preserved) | Single-file prefix rewrite (violates append-only; crash-unsafe) |
| D3 | Implementation contract | **Evolve the `Wal` trait** (`install_snapshot`/`load_snapshot`/`compact`; make `earliest` real); add Kernel `checkpoint`/`snapshot_shard`/`install_state` | Keep the inert `snapshot()` marker (cannot restore; blocks the milestone) |

### Binding requirements (maintainer, enforced by this milestone)

1. Frozen UCS/UCI documents remain unchanged.
2. Runtime interfaces may evolve **only** by activating previously reserved
   semantics (RT-001).
3. Kernel remains the **sole owner of truth**; Persistence never derives or
   interprets truth.
4. **Snapshot creation belongs exclusively to the Kernel**; Persistence stores
   **opaque snapshot bytes only**.
5. WAL remains **append-only**; compaction operates **only** by removing
   fully-covered segments.
6. Snapshot installation and recovery **must preserve deterministic replay**
   (`snapshot + tail replay` ≡ `full replay`, proven by test).
7. Recovery path becomes: `load_snapshot() → install_state() → replay(tail)`.
8. **No** architectural layers added; **no** ownership boundaries changed; **no**
   frozen specification documents modified.
